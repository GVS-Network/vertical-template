Context Glossary · v0.1.0

Words that mean specific things here.

Project-internal vocabulary. When Claude (or anyone) uses these terms in a PR or a prompt, they carry the definitions below — not the generic ones.

01 — Core terms

The vocabulary that drives every decision

Tenant

A business using the platform. In single-tenant mode (now) there is one tenant with tenantId === 'default'. In multi-tenant mode (phase 7) tenants are documents in the Tenant collection, resolved per-request from the host header. Maps directly to workspaceOwnerId in the agency app.

Vertical

A business type the platform is opinionated about. v1 ships four: bar-restaurant · food-truck · farm-source · screen-printer. Each has a preset.ts (SiteConfig defaults) and a theme.preset.ts (token defaults). A tenant declares its vertical; a tenant cannot occupy two verticals.

SiteConfig

The one TypeScript shape that describes what a build looks like. Lives twin in client/src/types/site-config.ts and server/src/types/site-config.ts, kept identical by contract:check. Contains tenantId · vertical · branding · features. Read at runtime through getSiteConfig(req).

Feature pack

A toggle-mounted module that contributes API + UI surface when enabled and zero surface when disabled. The five: catalog · content · intake · payments · auth (auth is technically middleware). Each has a folder on the server and a folder on the client. Routers conditionally mount on SiteConfig.features.<id>.enabled.

Preset

A starting set of values for a vertical. Two flavors: preset.ts (SiteConfig minus tenantId + branding) and theme.preset.ts (ThemeTokens minus tenant overrides). Consumed at init:vertical time only. Existing builds do not retroactively change when a preset changes — that is the Type-B propagation rule.

Seam

A single function or middleware boundary that the rest of the codebase reads through. The platform has two named seams: getSiteConfig(req) (tenant context) and getPaymentProvider() (payments). The point of a seam is that the body can change without callers changing.

Resolver

Specifically resolveTokens(tenantId) in the visual system. Walks the four theme layers (foundation → vertical → tenant pick → tenant override), validates contrast and font-weight requirements, emits the final CSS variable set. One function; the layers are data.

Registry

The set of named values (palettes, type pairings, block-style presets, etc.) that tenants pick from. The registry is platform-owned and grows deliberately — the visual brief's discipline: "register before second use." Tenants pick from the registry; they do not extend it.

Override

A tenant-supplied raw value that wins over a registry pick. Leaf-level only — color.brand.primary: '#7A1F1F' is an override; adding color.brand.tertiary is not, because it changes the shape. Overrides are validated; failed validations fail the build.

02 — Change taxonomy

Type A · Type B · Type C

From the template brief §05. These three letters carry a lot of weight in this project — use them precisely.

Type A
Core code

Feature packs, shared types, resolver, providers, middleware. Propagates via git pull. Client builds do not edit Type-A files; if they need different behavior, the change is generalized into core or rejected.

Type B
Presets & registry entries

Vertical presets, palette additions, pairing additions, block-preset additions. Available to opt into; existing builds keep their explicit picks unless they choose to migrate.

Type C
Per-build files

site.config.ts · site.theme.ts · Custom*.tsx · branded content. Never propagates. Once multi-tenant ships, Type C collapses into tenant documents.

03 — Phase vocabulary

What "phase" means in this project

Concept-brief phases (1–5) describe go-to-market stages — "pitch one-off builds personally," "backdoor the B2B layer." These are operator-business phases, not build phases.

Template-brief phase-2 refers specifically to the multi-tenant transition — the six-change jump documented in §06 of the template brief. This is what the build-docs call Phase 7.

Build-doc phases (0–7) are the prompts in docs/prompts/. They are the order in which code lands in the template repo. They map loosely to the concept brief's phase 1–2.

→

Disambiguate by context. "Phase 2" with no qualifier in a prompt means "build-doc phase 2" (feature packs). In a brief reference, "phase 2 of the concept brief" or "the template brief's phase-2 transition" — always say which.

04 — Implementation terms

Specific shapes and patterns

getSiteConfig(req)

The single server-side seam for tenant context. Today returns a singleton; in phase 7, returns per-request from req.tenantConfig. Application code calls it; nothing else.

PaymentProvider

Interface in server/src/providers/PaymentProvider.ts. Two adapters: stripe.ts, square.ts. Selected by SiteConfig. getPaymentProvider() returns the active one.

isFieldActive(field, formState)

The shared client/server predicate for conditional intake fields. Runs on the client for UX, runs on the server for enforcement. Form schemas live in SiteConfig.features.intake.forms[] — not in code.

resolveTokens(tenantId)

The visual system's resolver. Walks L1→L2→L3→L4, validates, emits CSS variables. Writes .theme-resolved.json alongside for auditability.

AssetRef

The shape of an uploaded asset (logo, etc.). Cloudinary-backed. Same pattern as the agency app's PortfolioProject uploads.

IntakeSubmission

The template's incoming-lead model. Shaped to match the agency app's Lead so phase 4 migration is a status-map + rename, not a model rewrite.

contract:check

The CI script that asserts client and server type files are byte-identical. Run on every PR. Without it, the dual-file convention drifts and the template brief's premise collapses.

05 — Terms intentionally avoided

What we don't call things

  • "Theme" alone — ambiguous. Say "theme tokens," "theme preset," or "resolved theme."
  • "Plugin" — feature packs are not plugins. Plugins suggest a third-party extension surface; feature packs are platform-owned and toggled, not installed.
  • "Workspace" in template code — the agency app's term, kept there. Template uses tenant. Mixing them collapses the phase-4 migration.
  • "Module" as a generic term — too overloaded. Use feature pack, provider, or middleware.
  • "Multi-tenant" applied to today's code — today is single-tenant. The seams support multi-tenancy; the system does not.
06 — Status

How this glossary updates

This is a working document. When a phase prompt introduces a new term that's specific to the project, that term lands here in the same PR. When a term changes meaning (or stops being used), it moves into §05 with a note explaining what replaced it.
