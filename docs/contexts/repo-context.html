The codebase family.

Three repositories, one lineage. What lives where, what each is for, and which conventions cross repo boundaries unchanged.

01 — The three repos

What exists, in one frame

Repo · ancestor baseapp
The MERN starter. Establishes every convention downstream — root-level client/ + server/, npm run dev with concurrently, Auth0 admin, Mongoose, Vite + React 18 + TS. Single-tenant. Generic. The thing both production repos were forked from.
Repo · production askanddeliverwebapp
The operator's own agency app. Already running. The de facto reference implementation for several thesis-critical patterns: workspace boundaries, the Lead model with conversion, Stripe Payment Links, express-oauth2-jwt-bearer, the SiteConfig Mongo model, Cloudinary-backed uploads. Where the new template makes a choice, the new template matches this app first.
Repo · in development vertical-template (working name)
The new platform template. Forked from baseapp, aligned with askanddeliverwebapp. Vertical-aware, multi-tenant-ready underneath, ships four vertical presets and a token registry. This is what every phase-N prompt is about.
i

Why two production repos, not one. The agency app stays the operator's ops backend through phase 4 of the concept brief — clients, projects, tasks, invoicing. The template ships the public surface and per-tenant ops. Phase 4 ports modules across. Trying to be both repos at once now means two copies of evolving code, which is the failure mode the template brief explicitly rules out.

02 — Inherited from baseapp

Conventions that cross unchanged

If a question about file layout, tooling, or scripts has a clean answer in baseapp, that is the answer. Do not invent a second convention.

Top-level layout

client/ and server/ at repo root. No monorepo tooling (no pnpm workspaces, no Turborepo). Each side is a self-contained Node project with its own package.json.

Dev loop

npm run dev at root runs concurrently against both sides. Hot reload on both. Single command, single terminal.

Client stack

Vite + React 18 + TypeScript. React Router for routing. No state library mandated — components and contexts unless the agency app shows otherwise for a specific concern.

Server stack

Express + TypeScript. Mongoose for Mongo. express-oauth2-jwt-bearer for Auth0 verification on admin routes. dotenv, structured logging, the usual.

Env files

Each side has its own .env. .env.example files committed; real .env ignored. Init scripts (phase 4) write into env files at the side that needs the value.

Identity

Auth0 short-term, per the concept brief's architecture decisions. Exit path planned to Clerk / WorkOS / self-host before scale. Treat express-oauth2-jwt-bearer as a wrapped dependency — application code never imports it directly outside the auth feature pack.

03 — Aligned with askanddeliverwebapp

Patterns lifted from the agency app

Anywhere the new template's choice could go several ways, it matches the agency app — because phase 4 of the concept brief is "operator becomes a tenant on the new system," which becomes a config exercise instead of a refactor only if the shapes already match.

Workspace boundary becomes tenantId

Agency app uses workspaceOwnerId on every model. Template uses tenantId. They are functionally identical and the migration is a field rename. Don't model anything that contradicts this isomorphism.

Lead model phase-4 portability

The template's IntakeSubmission mirrors the agency's Lead — notes with createdBy, conversion ids, pipeline status. When phase 4 ports the ops backend in, intake submissions become leads via a status-map.

Payments reuse, don't reinvent

Stripe Payment Links exactly as the agency app uses them. The PaymentProvider interface is new (Square is the second adapter), but the Stripe adapter's body should look familiar to anyone who's read the agency app.

SiteConfig (Mongo) separation of concerns

The agency app has a SiteConfig Mongoose model holding per-workspace branding and operational settings. The template's runtime SiteConfig type is the typed view of the same idea. Phase 7 will add a Tenant Mongoose model in the template that mirrors this shape.

Asset uploads single pipeline

Cloudinary, same configuration as the agency app's PortfolioProject uploads. Logo slots in the visual system's AssetRef resolve through this. When phase-2 lifts PortfolioProject into the template, the upload code already matches.

→

When in doubt: read the agency app first. If a pattern there solves the same problem, copy it. Diverging without a reason creates two ways to do one thing — and phase 4 will pay for the divergence in migration cost.

04 — Where the template diverges

What the new repo does that the others don't

✓

Vertical presets. The agency app is one shape; the template is four. Verticals live in verticals/<key>/ — site-config.preset.ts, seed/*.json, product-attributes.schema.ts, brand-stub.ts (tenant Type-C token overrides). Brand voice words live in theme/verticals/*.brief.md; L2 tokens in theme/verticals/*.tokens.ts (Phase 5). Provision with npm run init:vertical; local demo binds via BOUND_TENANT_ID in server/.env.

✓

Feature packs. Five packs (catalog · content · intake · payments · auth) under server/src/features/ and client/src/features/. Registry gates at boot (option b): disabled packs never import or register — routes 404. Pattern: server/src/features/_pack-pattern.md.

✓

Token registry. Four-layer theme resolution (foundation → vertical → tenant pick → tenant override). The agency app has theme constants; the template has a registry.

✓

Payment provider abstraction. Stripe + Square coexist behind one interface. Adding Toast or Clover later is one new file.

✓

Multi-tenant seams (phase 7). tenantId on every model from day one (single value 'default' for now). getSiteConfig(req) as the single seam. The six-change phase-2 jump is architected for.

06 — Cross-repo etiquette

What goes into which repo

baseapp

Frozen unless a convention truly changes. If a fix needs to apply to every downstream fork, it goes here. Otherwise it goes in the fork. Don't drift baseapp casually — the agency app pulls from it occasionally.

askanddeliverwebapp

Operator's production system. Changes here happen because the operator's business needs them. Patterns proven here graduate into the template when phase 4 arrives — but the agency app does not get experimental template features pushed back into it.

vertical-template

Where the platform work happens. Every phase-N prompt operates here. Client builds are forks (or branches, until phase 7) of this repo — see phase 6 for the per-build flow.

07 — Status

What Claude should assume when reading this context

baseapp and askanddeliverwebapp are live and already in production — they are reference material, not work surfaces, unless a phase prompt explicitly says otherwise. The vertical-template repo is the active build. When a phase says "match the agency-app pattern," that means: read the relevant files in askanddeliverwebapp, then implement equivalently in the template.

Things this context will need to update over time

Q · Repo URLs and paths

The actual git URLs and local clone paths for all three repos get filled in here once the template repo is created. Until then, references are by name only.

Q · Per-client repos

Q · Phase-7 collapse

When multi-tenant ships, per-client repos collapse into tenant documents inside the template. The §06 cross-repo etiquette table will need a fourth row for retired patterns.
