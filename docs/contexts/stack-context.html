01 — Core stack

What MERN means here

Runtime

Node 20 LTS. Use node --version matching .nvmrc at repo root.

Language

TypeScript on both sides. strict: true. No any without an inline justification comment. Prefer unknown + narrowing.

Client

Vite 5 · React 18 · React Router 6. No Next.js (Vite + React Router is the inherited choice; do not change without a brief revision).

Server

Express 4 · Mongoose 8 · MongoDB Atlas (managed). No Fastify, no Hono, no swap. express-oauth2-jwt-bearer for Auth0 JWT verification.

Identity

Auth0. Short-term decision per the concept brief. Wrap usage in the auth feature pack so the swap to Clerk/WorkOS is local.

Payments

Stripe (via Payment Links) and Square (with inventory). Behind PaymentProvider — application code never imports a vendor SDK directly.

Assets

Cloudinary. Same account/conventions as askanddeliverwebapp's PortfolioProject uploads.

Styling

CSS variables emitted by resolveTokens() into :root. No Tailwind (would bypass the registry). No CSS-in-JS for theme values. Component-scoped CSS modules are fine; their values reference the variables.

02 — Dev tooling

What runs locally

Root scripts (npm run …)

  • dev — concurrently runs client:dev + server:dev
  • build — production build of both sides
  • init:vertical -- --vertical=<id> --tenant=<slug> — scaffolds a new build from a preset
  • contract:check — verifies client/src/types/site-config.ts ≡ server/src/types/site-config.ts (and same for theme-tokens.ts)
  • doctor — env/health check; runs before dev
  • test — vitest, both sides
  • lint, typecheck — standard

CI gates (every PR)

  • typecheck · lint · test · contract:check
  • Theme validator runs on any site.theme.ts or theme.preset.ts change
  • Doc-maintenance check (see rules/documentation-maintenance.html)
03 — Allowed dependencies

What's in the toolbox, what to ask before adding

✓

Already approved. express · mongoose · react · react-router-dom · vite · typescript · zod · concurrently · express-oauth2-jwt-bearer · stripe · square · cloudinary · vitest · dotenv.

→

Requires a brief reference before adding. Any state library (Redux, Zustand), any UI kit (MUI, Chakra, shadcn), any ORM swap, any routing change, any styling-system change. The brief is your justification — if no brief covers it, write one or don't add it.

!

Forbidden without a thesis update. Tailwind · Next.js · server-side rendering frameworks · GraphQL · Prisma · monorepo tooling (Nx, Turborepo, pnpm workspaces) · any analytics SDK that adds runtime cost · any auth provider that isn't the planned Auth0 → Clerk/WorkOS migration target.

04 — Environment

What needs to be present for things to run

Required env vars (server)

  • NODE_ENV · PORT · MONGO_URI
  • AUTH0_DOMAIN · AUTH0_AUDIENCE
  • STRIPE_SECRET_KEY (when payments enabled with stripe)
  • SQUARE_ACCESS_TOKEN · SQUARE_LOCATION_ID (when payments enabled with square)
  • CLOUDINARY_URL
  • TENANT_ID — defaults to 'default' until phase 7

Required env vars (client)

  • VITE_API_BASE
  • VITE_AUTH0_DOMAIN · VITE_AUTH0_CLIENT_ID · VITE_AUTH0_AUDIENCE
  • VITE_TENANT_ID — mirrors server; phase 7 removes this in favor of runtime fetch

npm run doctor verifies presence and basic shape (URL formats, non-empty values). It does not validate credentials — that's dev's first request.

05 — Code conventions

House style for both sides

Imports

Absolute imports rooted at @/ on both sides (tsconfig paths). Group: external → @/types → @/features → local. No deep imports across feature-pack boundaries.

Files

kebab-case for files (site-config.ts). PascalCase for component files (IntakeForm.tsx). One default export per component file. Hooks live next to the components that own them, not in a global hooks/ folder.

Types

interface for object shapes that callers extend, type for unions and aliases. Schema validation via zod. Mongoose schemas always have a matching z.object() for inbound validation.

Async

async/await throughout. Errors caught at the route boundary, not deep. express-async-errors imported once in server/src/index.ts.

Tests

Vitest. Co-located *.test.ts next to the unit under test. Integration tests in tests/integration/ at the side where they run.

Comments

Comment why, not what. A reference to the brief that motivated a non-obvious choice (// see template-architecture-brief §03 — single seam for phase 2) is worth more than three lines explaining the code.

06 — Anti-patterns specific to this codebase

Things Claude should refuse

!

Editing core code from a client build. Per the template brief: client builds add Custom*.tsx files and override via site.config.ts / site.theme.ts. They do not edit feature packs, types, the resolver, or middleware.

!

Reaching past getSiteConfig(req). No route reads tenant context any other way. No process.env.TENANT_ID in route handlers. No direct config imports in feature packs. One seam, one entry.

!

Importing Stripe or Square SDKs outside server/src/providers/. The abstraction exists for a reason. If a route needs payment behavior, it calls getPaymentProvider().

!

Adding a model without tenantId. Every Mongoose schema declares it, indexed compound with its natural key. Single-tenant means the value is 'default', not absent.

!

Adding a token category outside the registry. Per the visual brief: leaf-level overrides only. A tenant can override color.brand.primary; they cannot introduce color.brand.tertiary. The category shape is platform-owned.

!

Hardcoding a CSS value that has a token. If --color-brand-primary exists, use var(--color-brand-primary). Hardcoded hexes outside theme/ are a build error.
