Context Session Starter · v0.3.0

Paste this at the top of every Cursor session.

A repeatable opening prompt that primes Claude with the briefs it should re-read, the phase the build is in, and the discipline that's currently in force.

01 — Why this exists

The cost of cold-starting every chat

Cursor auto-loads .cursorrules and matched .cursor/rules/*.mdc files. Those carry the standing rules. They do not carry: where this build is in the phase plan, which open questions are live, which client (if any) is the current build, and what is forbidden in this specific session.

The session starter fills that gap. It costs five seconds to paste and saves a half hour of corrections downstream.

02 — Before pasting

Fill in the four variables

The prompt below has four placeholders. Replace them before pasting; don't trust Claude to infer them.

{PHASE}

The build-doc phase you're working in. phase-0-bootstrap through phase-7-multi-tenant. If you're between phases (closing one, opening the next), state both.

{TEMPLATE_VERSION}

Current CHANGELOG.md top entry, e.g. v0.3.2. If unreleased, use v0.3.2-dev.

{CLIENT_BUILD}

If working on a per-client build (phase 6+), the tenant slug. Otherwise none — template work.

{OPEN_QUESTIONS}

One-line summaries of any live decisions not yet locked. Pulled from the briefs' or contexts' open-q sections that are currently relevant. If none, write none open.

04 — Variants

Short forms for specific cases

After-restart resumption

When Cursor restarts mid-phase, full reload is overkill. Paste this instead:

Resume · after Cursor restart, same phaseResuming the previous session. Phase: {PHASE}. Template: {TEMPLATE_VERSION}. Re-read docs/prompts/{PHASE}.html and the last 3 commits on this branch. Confirm the next prompt in that phase doc that we have not completed, then wait.

Client-build context

When working on a per-client build instead of the template itself (phase 6+):

Client build · per-tenant workThis session is per-client build work for tenant {CLIENT_BUILD}. Standing rule: we work in Type-C territory only. · Allowed: site.config.ts, site.theme.ts, Custom*.tsx files, branded assets. · Forbidden: any file under features/, providers/, theme/ (other than presets), middleware/, types/, getSiteConfig.ts, resolveTokens.ts. If you find yourself wanting to edit a forbidden file, STOP and tell me. The answer is one of: · the change goes into core (template PR, this client pulls the new tag) · the change is wrong (this client doesn't need it) · the rule needs revision (open question, escalate) Confirm and wait for the next prompt.

Brief-revision sessions

When the work is editing a brief, not writing code:

Thesis-edit session · brief revisionThis session is editing docs/{BRIEF}.html — a thesis revision, not implementation. Rules for this session: · No code changes. If you propose one, stop and remind me we're in thesis mode. · Bump the brief's version in the header. Bump docs/README.html version. · Add a CHANGELOG entry under "thesis" for the date. · Cross-reference: any other doc that depends on the section being edited must be flagged and either updated in the same PR or filed as a follow-up. Reason for revision: {REASON} Confirm and wait for the specific change.
05 — Hygiene

Keeping the starter honest

→

Update the starter when the doc list changes. Adding a context, retiring one, restructuring the prompts directory — all of these require editing this file. The doc-maintenance rule will warn you in CI.

→

Don't add per-session detail here. If the variable list grows beyond five or six fields, the starter is taking on responsibility that belongs in the phase prompts. Push detail down, not up.

i

You can save this as a Cursor "Composer template" or text snippet. The exact mechanism evolves with Cursor; the prompt body is what matters.

06 — Status

What this prompt expects of itself

v0.1.0 — the initial prompt and three variants (resume, client-build, brief-revision). Six required-reading docs. Four variables. The standing-rules block restates discipline already in .cursor/rules/ because rules and primes work better redundant than absent.

Updates over time

Q · Phase 7 starter

When multi-tenant ships, the standing rules add tenant-resolution boundaries and the "tenantId === 'default'" line gets edited or removed. A fourth variant for "tenant admin operations" may be needed.

Q · Skill-attached prompts

Recurring prompt patterns may graduate into docs/skills/. When that happens, the required-reading list grows and this file documents which skills auto-attach to which Cursor file globs.
