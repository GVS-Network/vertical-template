Template Architecture Companion to Concept Brief · v0.2

A vertical-aware MERN starter, multi-tenant-ready underneath.

Decisions, seams, and discipline for the new template that turns the platform thesis into something a build can actually start from on Monday.

01 — Lineage

Where this template sits in the codebase family

This template is not invented from scratch. It extends one repo and aligns with another — both already in active use, both already proving parts of the thesis.

Forked from baseapp
The MERN starter. Establishes conventions for everything downstream — root-level client/ + server/, npm run dev with concurrently, Auth0, Mongoose, Vite + React 18 + TS. The new template inherits all of these unchanged.
Aligned with askanddeliverwebapp
The agency app. Already generated from baseapp; already solved several problems the platform thesis needs — workspaces as tenant boundaries, a Lead model with conversion, Stripe Payment Links, express-oauth2-jwt-bearer, and a SiteConfig Mongo model. Where the new template makes choices, it matches the agency app first.
i

Why alignment matters. The concept brief's phase 4 is "operator migrates onto the new system as tenant #1." Aligning conventions now makes that merge a configuration exercise, not a refactor. The agency app's workspaceOwnerId is functionally the same construct as tenantId here — by phase 2, they become the same field.

02 — The one object

Every build is shaped by one SiteConfig

One TypeScript shape defines what a build looks like. It lives in two files (client/src/types/site-config.ts and server/src/types/site-config.ts) kept byte-identical by a contract-check script in CI.

interface SiteConfig {
  tenantId: string;
  vertical: 'bar-restaurant' | 'food-truck' | 'farm-source' | 'screen-printer';
  branding: { name, tagline, theme, fonts, contact };
  features: {
    catalog:  { enabled, mode, categories, squareInventorySync? };
    content:  { enabled, pages[] };
    intake:   { enabled, forms[] };           // forms have conditional fields
    payments: { enabled, provider, publicCheckout };
    auth:     { enabled, publicAccounts, provider };
  };
}
✓

Adding a capability the platform supports for everyone → core code change. Adding a choice a tenant can make about that capability → field on this type. The rule is the same rule from the concept brief, just made concrete.

03 — Decisions

Architectural choices, with the why

tenantId on every model phase-2 enablement

Every Mongoose schema declares tenantId indexed compound with its natural key. In single-tenant mode the value is always 'default'. The cost is zero today; the cost of retrofitting across N models on the day multi-tenant ships is high.

getSiteConfig(req) everywhere single seam for phase 2

Every server route reads its tenant context through one function. Today the function returns a singleton; in phase 2 it returns the per-request config. Zero callers change. This is the entire multi-tenant transition for application code.

Vertical presets, not forks discipline against drift

Each of the four verticals ships a preset.ts exporting a SiteConfig minus tenantId and branding. The init:vertical script writes one into env files at build init. Builds override via site.config.ts; they do not edit presets.

Feature packs are toggle-mounted modular updates

Six packs total — catalog, content, intake, payments, admin, plus auth as middleware. Each has a folder on the server and client. The router mounts them conditionally on SiteConfig.features.<id> (boolean flags). A disabled pack contributes zero API surface. admin requires auth.

PaymentProvider abstraction stripe + square coexist

A PaymentProvider interface with Stripe (via Payment Links, matching the agency app) and Square (with inventory fetch) adapters. The rest of the codebase calls getPaymentProvider() and never imports a vendor SDK. A third provider — Toast, Clover — is one new file.

Conditional intake = shared validator single source of truth

The same isFieldActive predicate runs on client (for UX) and server (for enforcement). Form schemas live in SiteConfig.features.intake.forms[], not in code. Adding a form for a tenant is config; adding a new field kind is core.

Intake mirrors agency's Lead phase-4 portability

The IntakeSubmission model has notes with createdBy, conversion ids, pipeline status — same shape as the agency app's Lead. A build that grows into the agency app's ops backend moves data with a status-map and a field rename, not a model rewrite.

Single-tenant first honest phasing

No Tenant model, no host-resolution middleware, no client config-fetch path, no tenant admin UI. All six are scoped in MULTI-TENANT-ROADMAP.md; none are built before there are multiple tenants to serve.

04 — Verticals

The four built-in presets, and what they encode

Each vertical pack is a starting point — opinionated about the parts that vary predictably by business type, silent about the rest.

bar-restaurant
Fixed-location food & drink

Bar, restaurant, café. Presence website + Square POS alignment + private-event intake. Catalog as menu (no checkout). The interesting choice: a structured menu instead of free text — so dishes can be sorted, hidden seasonally, rendered in multiple places.

catalog · menu
payments · square, staff-only
intake · private event w/ event-type conditional
publicAccounts · off
food-truck
Mobile food operation

Square POS like a restaurant, but a dynamic schedule and catering-forward intake. The conditional sales-guarantee question on festival/corporate bookings encodes domain knowledge a generic builder would never know to ask.

catalog · menu
payments · square, staff-only
intake · catering w/ guarantee conditional
publicAccounts · off
farm-source
Farm / CSA / farm-to-table

Most commerce-leaning vertical: real inventory, real public checkout, Square inventory sync on (small farms drift stock fast). The wholesale-inquiry form is the seed of the B2B network — its business_type field is shaped to absorb in-platform B2B traffic in phase 3.

catalog · inventory, square sync
payments · square, public checkout on
intake · wholesale w/ recurrence conditional
publicAccounts · on (CSA members)
screen-printer
Custom-print operation

Quote-driven, no off-the-shelf items. The only vertical defaulting to Stripe — because the workflow is quote→invoice→online payment, which is exactly what Payment Links exist for and exactly what the agency app already does. Long conditional intake; portfolio mode for the catalog.

catalog · portfolio
payments · stripe, staff-only
intake · project quote w/ garment-type conditional
publicAccounts · off
→

Two verticals share Square, two split. Bar-restaurant and food-truck cluster on Square because they cluster on a POS. Farm-source also takes Square because the operators already use it. Screen-printer takes Stripe alone because there's no POS counterpart. This is the brief's "food/Square cluster + creator/intake cluster" reflected in code.

05 — Modular updates

How template improvements reach client builds

The hardest problem in a productized-service model is upgrade flow. The template addresses it by classifying every change into one of three types — and being explicit about which types propagate cleanly.

Type A
Core code

Feature packs, shared types, payment abstraction, auth middleware, shared renderers. Propagates via git pull as long as client builds don't edit these files.

Type B
Vertical presets

Preset files. Consumed at init-time only — existing builds keep their current config. Advisory, not retroactive. Note in release notes; let each build opt in.

Type C
Per-build

Branding, content, intake schemas customized per client, hand-written client-specific pages. Never propagated. This is the intentional divergence.

The rule that makes it work

Client builds do not edit core code. If a client needs behavior the core doesn't provide:

  1. Tweak existing config via site.config.ts overrides — zero merge pain.
  2. Add a Custom*.tsx file alongside shared pages, mount in App.tsx below shared routes — one new file the upstream pull won't touch.
  3. Reject the request, or upstream it. If a one-off looks like a second client's need, generalize now.
→

Builds pin to release tags, not branches. Bumping a client from v0.4.1 to v0.4.2 is one git pull and a smoke test. Once multi-tenant ships in phase 2, the whole Type-C column collapses — there is no per-build divergence anymore.

06 — The phase-2 jump

From single-tenant to multi-tenant in six discrete changes

The architecture buys back this transition deliberately. Six files (or so) touched. No feature pack, no model, no vertical preset, no page changes.

01Model

Add a Tenant Mongoose model

Shape mirrors SiteConfig: { _id (slug), vertical, branding, features, host }. Why separate from runtime SiteConfig: keeps a typed boundary so migrations can evolve the document while runtime code stays stable. Same separation the agency app uses for its SiteConfig.

02Middleware

Tenant-resolution middleware

One file: read host header, look up Tenant, attach to req.tenantConfig. Mounted globally before feature routers. Mirrors the agency app's getWorkspaceOwnerId pattern.

03Swap

Swap getSiteConfig's body

return req.tenantConfig ?? loadSiteConfig(). No callers change. The fallback covers cron jobs and tests.

04SSL

Custom-domain SSL provisioning

Caddy in front of the app (Let's Encrypt on-demand for registered hosts) or Vercel Domains API. Deployment surface change, not application code.

05Client

Client config from /api/_meta/config

The endpoint already exists. Client swaps build-time env-based loading for a runtime fetch. Largest of the six changes — budget a day.

06Admin

Tenant admin UI

New /admin/tenants section, operator-only. Until self-service exists, the operator IS the tenant admin — exactly the staged go-to-market from the concept brief.

07 — Honest scope

What this template deliberately does not include

i

Tenant admin ships in build-doc Phase 6. The admin feature pack provides /admin for business owners (pages, events/posts, catalog, submission inbox). Content write APIs ship in the same phase. Platform operator admin (/__admin/tenants, fleet ops) remains build-doc Phase 8 — not the same surface.

i

No PortfolioProject model. When a build needs portfolio (screen-printer especially), bring in the agency app's PortfolioProject model wholesale. Cloudinary uploads and mixed-media galleries are already solved there.

i

No ops backend. Clients, projects, tasks, time entries, invoicing — these live in the agency app and stay there. Phase 4 of the concept brief brings them in by porting modules from askanddeliverwebapp. Doing it now would mean maintaining two copies of evolving code.

i

No multi-tenant identity. The seams are in place. The Tenants collection and per-request resolution land in phase 2 — six discrete changes, scoped above.

08 — Status

What the template ships today

A working scaffold: 4 vertical presets, 5 feature packs (plus auth; admin lands Phase 6), a contract-check script, a doctor script, an init-vertical script, complete docs (architecture, feature packs, verticals, multi-tenant roadmap, upgrade flow, per-vertical), and a payment-provider abstraction with Stripe and Square adapters wired up to the agency-app patterns.

Immediately ready to do

  • Pick a vertical and run npm run init:vertical -- --vertical=<id> --tenant=<slug>
  • Edit site.branding.ts on both sides
  • Wire Auth0 (uncomment in middleware/auth.ts) and the payment SDK of choice
  • Start building the public surface and admin UI specific to the client

Open questions deferred to subsequent threads

  • Custom-domain SSL strategy — Caddy vs. Vercel for phase-2 multi-tenancy.
  • File uploads on intake — when a second vertical needs them, lift Cloudinary-backed upload from the agency app into core.
  • Admin UI patterns — how much of the agency app's admin shell is portable into vertical builds vs. how much is intentionally simpler per vertical.
  • Phase-2 trigger — what N tenants and what feature stability constitutes the green light.
