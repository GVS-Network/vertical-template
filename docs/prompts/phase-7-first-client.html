PHASE 7 First Client · v0.7.0 → v0.8.0

Phase 7 — First client.

Drive one real client engagement through the template end-to-end. Operator surface (Phase 6) is already in core — this phase is custom public frontend, real content, brand picks, and production cutover. Planned first client: Terrible Gerald's (food-truck).

01 — Prerequisites

Before you start this phase

  • Phase 6 closed, tagged v0.7.0. Tenant admin, content writes, intake email, Markdown rendering.
  • You have a real client signed — not a hypothetical. The whole point of phase 7 is realness on a live domain.
  • The client falls into one of the four verticals. (If they don't, defer them to a later phase; don't invent a fifth vertical mid-build.)
  • You have a content intake mechanism with the client — even if it's a shared doc. The template can't render what doesn't exist.

Phase 7 is a single-tenant deploy. We are NOT introducing multi-tenant routing yet. This client gets their own deploy of the template (forked or branched), bound via BOUND_TENANT_ID on its own DB. Phase 8 collapses multiple deploys into one.

Discoveries become brief revisions, not template hacks. When the client wants something the template can't do cleanly, the first response is "is this a brief problem or a template problem?" — not "let me add a special-case branch".

03 — Change taxonomy in this phase

What kind of changes are we making?

Type A
Client tenant files

Creating theme/tenants/<client>.pick.ts and seed content is additive. No existing code path changes. This is the right kind of change for a client-specific phase.

Type B
(careful here)

If the client needs a token that doesn't exist, the brief discipline kicks in. Either escalate to vertical-tokens (changes everyone), or add a leaf to ThemeTokens (Type B — ripples to every layer). Do not silently inline.

Type C
Discovered structural needs

If the client surfaces a need for a structural template change (e.g. orders need a "scheduled pickup date"), it's Type C. Document, propose, decide whether to do it now (delays launch) or after (carry as debt with explicit deadline).

04 — Prompts to run, in order

The phase-7 prompt sequence

Phase 7 mixes prompts with operator-only work (talking to the client, picking a domain, etc). Prompts are flagged CLAUDE or OPERATOR. Do not rebuild admin, content writes, or intake email — Phase 6 shipped those.

Prompt 7.1 OPERATOR — Sign the client + write a project brief

Prompt 7.1 — operator task Before any prompt to Claude, you (the operator) must: 1. Have the client signed (or have explicit OK to use them as the phase-7 test). 2. Write a one-page project brief: client name, vertical, what they sell, who their customers are, what content they have ready, what payment provider they want, what domain, what launch date. 3. Confirm the client's vertical IS one of the four. If not, STOP — do not invent a fifth vertical here. Save the brief at build-docs/clients/<client>.brief.md. Every following prompt assumes it exists.

Prompt 7.2 CLAUDE — Stand up the client's instance

Prompt 7.2 We have a real client at build-docs/clients/<client>.brief.md. Read it. This is a single-tenant deploy. Steps: 1. Confirm the client's vertical key matches a registered preset. 2. Decide infrastructure path: separate deploy (current state), OR shared multi-tenant deploy (NOT YET — Phase 8). For phase 7, separate deploy. 6. Enable features.auth and features.admin in site config; payments per brief (TG: off). 3. Choose a tenant id (kebab-case, no reserved words). 4. Run init:vertical -- --preset=<v> --tenant=<client-tenant-id> against a NEW database (not the dev one). 5. Walk the seeded app for that tenant. Note any seed content that needs replacing with the client's actual content. Report what was created and what content is still placeholder.

Prompt 7.3 OPERATOR — Content intake

Prompt 7.3 — operator task Get real content from the client. Bare minimum: - Real product/menu list with real prices. - Real "About" copy (1-3 paragraphs in their voice). - Real photos (or commit to using stock as a stopgap, with replacement timeline). - Real contact info, hours, location. - Logo file (vector if possible). Put it all in a structured form Claude can read: a single content.intake.md or a folder under build-docs/clients/<client>/content/. Until this is done, no further Claude prompts. The template can't render fake content into a real launch.

Prompt 7.4 CLAUDE — Load the real content

Prompt 7.4 We now have real content at build-docs/clients/<client>/content/. 1. Replace the seeded Products in the client's tenant with the real product list. 2. Update the Page records (Home, About, etc) with real copy. 3. Upload logo + photos to whatever asset host we're using (per stack-context). 4. Update the FormDefinition for "Contact" or equivalent. For each replacement, log what was placeholder before and what is real now. Do NOT change any code. Only data. If any required content is missing, list it (don't hallucinate replacements). Commit: "phase-7: real content for <client>".

Prompt 7.5 CLAUDE — Custom public frontend

Prompt 7.5 Build Type C custom pages (CustomLayout, CustomHome, CustomEvents, etc.) wired to platform APIs — not Storyblok/Getform. Migrate assets from customers/<client>/ reference. Match majority of existing styling. Commit: "phase-7: custom frontend for <client>".

Prompt 7.6 CLAUDE — Real brand picks

Prompt 7.6 The client has visual preferences. Create theme/tenants/<client>.pick.ts based on those. Process: 1. Read theme/verticals/<v>.tokens.ts to see what's vertical-default for this client. 2. List what the client wants different (from the brief or from a call). 3. For each difference: is it expressible as a curated pick (a named option we offer) or does it need a leaf override? - If pick: add it to the pick file. - If override: add it to <client>.override.ts, BUT only if it's a single leaf. If it's structural, FLAG IT and we'll decide whether to escalate or refuse. 4. Run the contrast validator against the resolved tokens. If it fails, the client's preference is breaking accessibility — surface that with options. 5. Render the site and screenshot the home page. Save the screenshot to build-docs/clients/<client>/screenshots/. Commit: "phase-7: brand picks + (maybe) leaf overrides for <client>".

Prompt 7.7 CLAUDE — Pre-flight checks

Prompt 7.7 Before any deploy, run a pre-flight pass against the client's tenant: 1. doctor.ts — must pass with the client's env. 2. contract:check — must pass. 3. contrast validator on resolved tokens for <client> — must pass. 4. Walk every page in the rendered app, log any 404, 500, broken image, or "lorem". 5. Confirm payment provider if features.payments (skip for TG). Intake test submission delivers email. 6. Confirm Auth0 callback URLs for client domain; client can log into /admin and edit one page. Produce a pre-flight report. Do not proceed if any check fails.

Prompt 7.8 OPERATOR — Deploy to staging

Prompt 7.8 — operator task Deploy the client's instance to a staging URL (e.g. clientname.staging.<your-domain>.com). SSL configured. Walk the client through it. Capture their feedback in build-docs/clients/<client>/feedback-staging.md. Iterate via content/pick adjustments or scoped Type C frontend fixes. Template-level discoveries → 7.11.

Prompt 7.9 OPERATOR — Production cutover

Prompt 7.9 — operator task Production cutover. Steps: 1. Point the client's real domain at production infra. 2. Issue SSL certificate. 3. Switch payment provider from test to live mode. Update STRIPE_SECRET_KEY / SQUARE_ACCESS_TOKEN, etc. 4. Update Stripe / Square webhook URLs to the live domain. 5. Run doctor in production mode — confirm webhook signatures verify against live keys. 6. If payments on: smoke-test $1 purchase + refund. If off: smoke-test intake form + admin edit. Commit any env or runbook changes to build-docs/clients/<client>.md. Tag: v0.8.0-client-<name>.

Prompt 7.10 CLAUDE — Write the runbook

Prompt 7.10 Create build-docs/clients/<client>.md as a runbook. Sections: - Tenant info (id, vertical, deploy URL, infra notes). - Payment provider details (which one, account id, where to refund, where to view payouts). - Content edit workflow (where pages live in the admin, who has access). - Common operations: add a product, change hours, post an update, replace a photo. - Troubleshooting: known false alarms, where logs are, how to escalate. - Contact: who at the client to call, who internally owns this account. This is the doc the operator (or a teammate) will open at 9am Monday when the client emails about a typo. Make it scannable.

Prompt 7.11 CLAUDE — Discoveries log + brief revision proposals

Prompt 7.11 Throughout phases 7.1-7.10 you (and the operator) have noticed things. Write them up. Create build-docs/discoveries/<client>-phase-7.md with sections: ## What worked as the briefs predicted ## What was easier than expected ## What was harder than expected ## What surprised us ## What contradicted a brief ## Recommended brief revisions (one per contradiction) For each "recommended brief revision" propose: - Which brief (concept / template / visual-system). - What sentence/section changes. - Why this change does not weaken the brief's thesis. Do NOT change any brief yet. Proposals are reviewed by the operator before any brief is edited.

Prompt 7.12 CLAUDE — Close the phase

Prompt 7.12 Doc update pass: - Append Phase 7 entry to CHANGELOG. Client name, vertical, brand picks, discoveries. - Bump package.json to 0.8.0. - Update repo-context.html §05 for client artifacts. - List OPEN_QUESTIONS for Phase 8. Tag v0.8.0 and v0.8.0-client-<name>.
05 — Acceptance criteria

Phase 7 is done when…

  • Client site live at real domain with SSL.
  • Client owner can use /admin without developer help.
  • Intake form emails client contact address; submissions visible in admin inbox.
  • Payment live-mode + real transaction only if features.payments is on.
  • theme/tenants/<client>.pick.ts (+ optional override) populated.
  • Runbook and discoveries log complete.
  • Tagged v0.8.0 and v0.8.0-client-<name>.
