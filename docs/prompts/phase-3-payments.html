PHASE 3 Payments · v0.3.0 → v0.4.0

Phase 3 — Payments.

Implement getPaymentProvider behind one interface, with two providers: Stripe (Payment Links, no PCI surface) and Square (with inventory sync). Webhooks reconcile orders. The payments feature pack stays exactly as it was — only the seam fills in.

01 — Prerequisites

Before you start this phase

  • Phase 2 closed, tagged v0.3.0. All five packs pass pattern audit.
  • The payments pack mounts cleanly and fails checkout intent with the expected NotImplementedYet error.
  • You have both a Stripe test account and a Square sandbox account set up. Don't start this phase without them.
  • You have read the Stripe Payment Links docs and Square Orders/Inventory docs at least once recently.

Real money, real care. This phase touches the only code path in the template that can lose customer money. Idempotency, webhook signature verification, and replay protection are non-negotiable. Don't move fast through prompts 3.5 and 3.6.

02 — What this phase produces

Deliverables

The PaymentProvider interface (final)single contract

From Phase 1's stub: createCheckout(order), verifyWebhook(req), parseWebhookEvent(req), plus optional syncInventory(products). Optional methods return NotSupported in providers that can't.

Stripe providerPayment Links flow

Uses Stripe Checkout Payment Links. We do not collect card data. Order → Payment Link → redirect → webhook → markOrderPaid. No PCI surface in the template.

Square providerwith inventory

Uses Square Orders API + Checkout. Implements optional syncInventory to push stock changes into Square's catalog (since Square is many small businesses' POS source of truth).

Webhook receiverssignature-verified, idempotent

POST /api/payments/webhook/stripe and /square. Verify signatures. Deduplicate by event ID. Update orders idempotently. Log every event.

getPaymentProvider implementationseam closes

Reads siteConfig.payment.provider; returns the right concrete provider. Throws a recognizable error if 'none' and a payments-pack route is called.

doctor.ts growsenv checks

If payments feature flag is on, doctor checks that the relevant provider's env vars are present and that a test call to the provider's API succeeds.

What kind of changes are we making?

04 — Prompts to run, in order

The phase-3 prompt sequence

Prompt 3.1 — Carry forward open questions

Prompt 3.1 Resolve OPEN_QUESTIONS from Phase 2's close. Then add to that list: - Do we want to support BOTH providers simultaneously per site (e.g. Stripe for online, Square for in-person), or is it one-per-site? - For Square, do we MUTATE Square inventory or only READ it? The brief implies sync; pick a direction and document. Resolve both before any code. Document the answers in features/payments/README.md.

Prompt 3.2 — Lock the PaymentProvider interface

Prompt 3.2 Open server/seams/get-payment-provider.ts. Replace the stub with the final interface: interface PaymentProvider { key: 'stripe' | 'square'; createCheckout(order: Order): Promise<{ url: string; providerRef: string }>; verifyWebhook(req: Request): boolean; parseWebhookEvent(req: Request): WebhookEvent; syncInventory?(products: Product[]): Promise<void>; } type WebhookEvent = | { type: 'order.paid'; providerRef: string; eventId: string } | { type: 'order.failed'; providerRef: string; eventId: string } | { type: 'order.refunded'; providerRef: string; eventId: string } | { type: 'unknown'; eventId: string }; Leave getPaymentProvider() body still throwing — providers come next. Commit: "phase-3: lock PaymentProvider interface".

Prompt 3.3 — Stripe provider implementation

Prompt 3.3 Create features/payments/providers/stripe.ts implementing PaymentProvider. - createCheckout(order): create a Stripe Checkout Session using line items derived from order.items. Return { url, providerRef: session.id }. - verifyWebhook(req): use stripe.webhooks.constructEvent with STRIPE_WEBHOOK_SECRET; return boolean. - parseWebhookEvent(req): map Stripe events: checkout.session.completed → order.paid checkout.session.expired → order.failed charge.refunded → order.refunded (everything else) → unknown - syncInventory: NOT implemented. Throw NotSupported. Env additions: STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET. Wire it into getPaymentProvider when siteConfig.payment.provider === 'stripe'. Test the full happy path against Stripe TEST mode: create an order, hit /intent, get a URL, complete checkout, watch the webhook fire, confirm order.status flips to 'paid'. Commit: "phase-3: Stripe provider + happy-path test".

Prompt 3.4 — Square provider implementation

Prompt 3.4 Create features/payments/providers/square.ts implementing PaymentProvider. - createCheckout(order): create a Square Order, then a Checkout link. Return { url, providerRef: orderId }. - verifyWebhook(req): verify Square's HMAC signature using SQUARE_WEBHOOK_SIGNATURE_KEY. - parseWebhookEvent(req): map Square events: order.updated (state PAID) → order.paid order.updated (state CANCELED) → order.failed refund.created → order.refunded (else) → unknown - syncInventory(products): for each product with sku, push stock to Square Catalog Inventory API. Idempotent. Env additions: SQUARE_ACCESS_TOKEN, SQUARE_LOCATION_ID, SQUARE_WEBHOOK_SIGNATURE_KEY, SQUARE_ENV ('sandbox'|'production'). Wire into getPaymentProvider when siteConfig.payment.provider === 'square'. Test against Square SANDBOX: same happy path as Stripe, plus run syncInventory against one product and confirm stock appears in Square dashboard. Commit: "phase-3: Square provider + inventory sync".

Prompt 3.5 — Webhook handlers (with idempotency)

Prompt 3.5 THIS IS THE MONEY-CRITICAL PROMPT. Read it carefully and do not skip steps. Update features/payments/controller.ts webhook routes: POST /api/payments/webhook/:provider 1. Look up provider via getPaymentProvider on a synthesized siteConfig matching :provider (since webhooks don't carry tenant). For phase 3, this means we accept the param as the lookup key; phase 7 will revisit. 2. Call provider.verifyWebhook(req). If false → 400 + log. 3. Call provider.parseWebhookEvent(req). 4. Check WebhookEventLog (new model: tenantId, provider, eventId — unique index on (provider, eventId)). If already seen → 200 (idempotent acknowledge), do nothing else. 5. Insert WebhookEventLog row. 6. Apply event: order.paid → service.markOrderPaid(providerRef, eventId) order.failed → service.markOrderFailed(providerRef) order.refunded → service.markOrderRefunded(providerRef) unknown → just log 7. Return 200. Test: - Send the same Stripe event twice. Confirm order moves to paid exactly once. - Send a tampered signature. Confirm 400. - Send an unknown event type. Confirm 200 and a log entry. Commit: "phase-3: webhook handlers with idempotency + signature verification".

Prompt 3.6 — Doctor checks for payments

Prompt 3.6 Extend scripts/doctor.ts: - If features.payments is true on defaultSiteConfig: - If payment.provider === 'stripe': require STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET. Hit a trivial Stripe API endpoint to confirm credentials. - If payment.provider === 'square': require SQUARE_ACCESS_TOKEN, SQUARE_LOCATION_ID, SQUARE_WEBHOOK_SIGNATURE_KEY, SQUARE_ENV. Hit a trivial Square API endpoint. - If payment.provider === 'none' but the pack is on: print a WARN (not a fail). Confirm doctor still passes when payments feature is off. Commit: "phase-3: doctor checks for payment providers".

05 — Acceptance criteria

Phase 3 is done when…

  • End-to-end Stripe TEST checkout works: order → /intent → URL → pay → webhook → status: 'paid'.
  • End-to-end Square SANDBOX checkout works, same flow.
  • Sending the same webhook twice flips the order to paid exactly once.
  • A tampered webhook signature returns 400.
  • syncInventory works for Square; throws NotSupported for Stripe.
  • doctor verifies provider creds when payments is on.
  • Tagged v0.4.0.

Both providers, both green, both webhooks idempotent. This is the single best signal that the seam pattern from Phase 1 was worth it. If filling it in here was painful, that's information — note it for the brief revision.
