PHASE 2 Feature Packs · v0.2.0 → v0.3.0

Phase 2 — Feature packs.

Build five self-contained, toggle-mounted feature packs: catalog, content, intake, payments, auth. Each ships schema + service + controller + router + client components, owns its own folder, reads req.siteConfig.features.* at the door, and stays invisible when its flag is off.

01 — Prerequisites

Before you start this phase

  • Phase 1 is closed, tagged v0.2.0. contract:check passes.
  • req.siteConfig is populated and at least one route reads from it.
  • Every Mongoose model has tenantId; the scoped() helper works.
  • You have decided how feature toggles gate routes (resolved from Phase 1's open question).

One pack at a time. Resist the urge to scaffold all five in one go. Each pack lands as its own commit with its own checks. If pack #2 reveals a flaw in the pack pattern, you want to fix it once before stamping it five times.

02 — What this phase produces

Deliverables

The pack patterndocumented before any pack is built

server/src/features/_pack-pattern.md — folder shape, exports, registry gate, tenant rules. Every pack conforms.

features/catalogproducts + inventory shape

Schema for Product (name, sku, price, stock, vertical-flexible attributes), service layer, REST controller, mounting router. Client list and detail components.

features/contentpages, posts, media

Page + Post schemas (slug, body, status, hero). MDX-or-rich-text-store decision documented. Client: page renderer, post list.

features/intakeforms & submissions

FormDefinition + Submission schemas. Generic field renderer on client. Email-or-store-only decision documented per vertical's needs (defer email integration to a later pack-internal phase if needed).

features/paymentsscaffold only, no provider

Order schema, checkout intent endpoint, webhook receiver shell. All provider-specific code stays in features/payments/providers/* — empty in phase 2. Phase 3 fills these.

features/authAuth0 wiring

Per stack-context, Auth0 via express-oauth2-jwt-bearer. Pack exports a requireAuth middleware. Client has a login button + a useSession hook. Tenant scoping respected.

Toggle-mountingthe central invariant

A server/features/registry.ts reads siteConfig.features and mounts only the packs that are on. Off packs do not run middleware, do not register routes, do not load their client JS.

03 — The pack pattern

What every pack looks like

Folder shape

Each pack lives under server/features/<pack>/ on the server and client/features/<pack>/ on the client. Inside each:

SERVER SIDE — features/<pack>/

1
schemas/ — Mongoose models. Each includes tenantId and uses scoped() for all queries.
2
service.ts — Pure functions over schemas. No req/res. Easy to test in isolation.
3
controller.ts — Express handlers. Thin. Calls service, returns JSON.
4
router.ts — Express router factory; only called when the registry has already enabled the pack (no null-router gate — see 2.1 resolutions).
5
index.ts — Pack entrypoint: export default {{ key: 'catalog', register(app, siteConfig) }}.

CLIENT SIDE — features/<pack>/

1
components/ — Pack-owned components. No global selectors; everything namespaced.
2
hooks/ — Data hooks calling the pack's API. Use a shared fetcher.
3
routes.tsx — Route definitions to be merged into the app router only when the feature flag is on.
4
index.ts — Exports routes and any host-facing components other packs might compose.

05 — Acceptance criteria

Phase 2 is done when…

  • All five packs exist under server/features/ and client/features/.
  • Toggling any feature in defaultSiteConfig.features off cleanly removes that pack's routes (404, not 500).
  • No pack imports from another pack. Packs talk via SiteConfig, seams, or shared types/ only.
  • Every pack's schema is tenantId-scoped and every query uses scoped().
  • The payments pack mounts when toggled on but cleanly fails the checkout intent with a recognizable "phase-3 needed" message.
  • Tagged v0.3.0.

You now have a horizontal MERN with five packs. This is the most reusable artifact in the whole template. Even if vertical SaaS as a thesis pivots tomorrow, what you've built in phase 2 is independently valuable.
