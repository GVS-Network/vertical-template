PHASE 8 Multi-Tenant · v0.8.0 → v1.0.0

Phase 8 — Multi-tenant.

The six discrete changes that turn a single-tenant template into a multi-tenant platform. The template-architecture brief promised this would be six changes, not sixty — Phase 1's tenantId investment and Phase 5's tenant-keyed token resolution are what cash that promise. After this phase, one deploy serves many sites.

01 — Prerequisites

Before you start this phase

  • Phase 7 closed, tagged v0.8.0. A real client is live single-tenant.
  • The tenant-leakage test from Phase 4 still passes (different BOUND_TENANT_IDs show no cross-talk).
  • You have read the template-architecture brief's "phase-2 jump" section (which is this phase under the build doc's numbering) and counted the six changes for yourself.
  • You have a credentials story for SSL at scale — either a wildcard, a per-domain auto-issuer (e.g. Caddy/Let's Encrypt), or a managed front (e.g. Cloudflare). Decide before prompt 7.4.

The phase-7 client must not break. Their single-tenant deploy is in production. Phase 8's success criterion is "we collapsed N deploys into 1 and no customer noticed". Do not skip the migration verification at the end.

02 — The six changes

What this phase produces — exactly

From the template-architecture brief: this jump is exactly six discrete changes. If you find yourself making a seventh, stop and ask whether it's actually part of one of these six, or whether it's scope creep that should be its own later phase.

Change 1 — Tenant modelfirst-class entity

Introduce a proper Tenant Mongoose model (replacing the placeholder _tenants collection from Phase 4). Fields: _id (tenantId string), preset, siteConfigOverrides, hosts (array of hostnames), status, createdAt, updatedAt.

Change 2 — Host middlewarerequest → tenant

Express middleware that reads req.hostname, looks up the matching Tenant, attaches it to req.tenant. Runs before req.siteConfig is set.

Change 3 — getSiteConfig swapthe seam pays off

getSiteConfig(req) body changes from return defaultSiteConfig to return mergeDefaults(req.tenant.siteConfigOverrides). Zero callers change. The signature has been right since Phase 1.

Change 4 — SSL strategyinfra, not code

Implement whatever was decided in prereqs: wildcard cert, auto-issuance per host, or a managed front. Code-side this is small (maybe a host-allowlist check); infra-side it's the bulk of the work.

Change 5 — Tenant config adminhow operator manages tenants

A protected admin UI at /__admin/tenants: create a tenant, list tenants, edit a tenant's siteConfigOverrides and hosts, suspend a tenant. Auth0 with an operator role.

Change 6 — Ops dashboardvisibility

A read-only dashboard at /__admin/ops: per-tenant request counts, recent errors, payment-provider health, last successful inventory sync. The thing the operator looks at at 9am Monday.

03 — Change taxonomy

What kind of changes are we making?

Type A
Tenant model fields

Adding the Tenant model is additive — nothing references it until host middleware activates. Pure Type-A creation.

Type B
(none in this phase)

No visual-token work happens here. Tokens already resolve by tenantId; multi-tenant just changes how tenantId arrives.

Type C
getSiteConfig body swap

Even though the signature doesn't change, the body change is Type C because it touches the contract every caller relies on. Test thoroughly. Stage carefully.

04 — Prompts to run, in order

The phase-8 prompt sequence

Prompt 8.1 — Carry forward + audit Phase 7 surprises

Prompt 8.1 Resolve OPEN_QUESTIONS from Phase 7. Pay special attention to the closing question: did the phase-7 client need anything that tenantId-on-every-model didn't buy? If YES — there is a missing tenantId scope somewhere — fix it FIRST, before any Phase 8 work. Phase 8 assumes every cross-tenant boundary is clean. Audit: 1. Re-run the tenant-leakage test from Phase 4 with three tenants now (default, demo, phase-7 client codename). 2. Check every webhook handler: does it correctly route to the right tenantId based on something other than req.hostname? (Webhooks don't carry our hostname.) 3. Check every cron/background job (if any): is it iterating by tenantId, not assuming a single tenant? Report the audit. Do not start prompt 7.2 until clean.

Prompt 8.2 — Change 1: Tenant model

Prompt 8.2 Create the Tenant Mongoose model at server/models/tenant.ts. Schema: _id: String (this IS the tenantId — no separate id field) preset: 'bar-restaurant' | 'food-truck' | 'farm-source' | 'screen-printer' siteConfigOverrides: Mixed (partial SiteConfig, merged onto defaults at request time) hosts: [String] (array of hostnames this tenant answers to; e.g. ['client.com', 'www.client.com']) status: 'active' | 'suspended' | 'archived' paymentProviderRef: String (provider's account id, for routing webhooks) createdAt, updatedAt Indexes: - unique on _id (Mongoose default for _id). - sparse unique on each element of hosts (a host can serve at most one tenant). Migrate existing _tenants placeholder docs from Phase 4 into the real Tenant collection. Write a migration script at scripts/migrate/0001-tenants.ts. Idempotent. Commit: "phase-8: change 1 — Tenant model + migration".

Prompt 8.3 — Change 2: Host middleware

Prompt 8.3 Create server/middleware/resolve-tenant.ts: - Read req.hostname. - Look up Tenant.findOne({ hosts: hostname, status: 'active' }). - If found: req.tenant = tenant. - If not found: - In development with BOUND_TENANT_ID set: load that tenant by id. - Otherwise: respond 404 with a "no tenant for this host" page (NOT a server error — these are valid lookups that miss). Mount this middleware AS EARLY AS POSSIBLE in the chain, before any router or auth, but AFTER cookie/body parsing. Add types/express.d.ts augmentation for req.tenant. Test: - Hit the app with a recognized host: req.tenant populated. - Hit with an unknown host: clean 404. - Hit a suspended tenant: clean 404 (or 503 with a "suspended" page — decide and document). Commit: "phase-8: change 2 — host middleware (resolve-tenant)".

Prompt 8.4 — Change 3: getSiteConfig swap (THE big one)

Prompt 8.4 This is the change that pays off the Phase 1 seam investment. Open server/seams/get-site-config.ts. Change the body: export function getSiteConfig(req: Request): SiteConfig { if (!req.tenant) throw new Error('getSiteConfig called before resolve-tenant middleware'); return mergeDeep(defaultSiteConfig, req.tenant.siteConfigOverrides); } The signature is unchanged. Therefore no other code in the repo needs to change. If you find yourself wanting to change a caller, STOP — that's a sign the seam was wrong or this change is. After the swap: 1. Run all tests. 2. Boot with two seeded tenants on different hosts (use /etc/hosts locally to fake it). 3. Confirm each host renders its own tenant's config — branding, features, payment provider. 4. Confirm req.siteConfig.tenantId === req.tenant._id everywhere. Commit: "phase-8: change 3 — getSiteConfig reads req.tenant".

Prompt 8.5 — Change 4: SSL strategy

Prompt 8.5 Per the SSL decision made in prereqs, implement and document. If wildcard cert (subdomain-only model): - Update DNS docs in build-docs/clients/<client>.md to reflect "your-name.<our-domain>.com is how you get on platform". - No code changes beyond a host-allowlist check. If auto-issuance per host (Caddy / autocert / cert-manager): - Stand up the auto-issuer in front of the app. - Add a host-allowlist in code: only attempt issuance for hosts that map to an active Tenant. If managed front (Cloudflare, Fastly, etc): - Configure the front with the platform's origin. - Document the manual step of adding each new host to the front. Update scripts/doctor.ts: in production, fetch https://<tenant.hosts[0]> from the running app's perspective and verify a 200 with valid cert. Fail loud if not. Commit: "phase-8: change 4 — SSL strategy (<strategy-name>)".

Prompt 8.6 — Change 5: Tenant admin UI

Prompt 8.6 Build an admin UI at /__admin/tenants. It is gated by Auth0 with a role check (only users with the "operator" role can access). Routes: GET /__admin/tenants — list all tenants GET /__admin/tenants/:id — show one tenant + its config overrides + hosts POST /__admin/tenants — create new tenant (preset + initial host) PUT /__admin/tenants/:id — edit overrides, hosts, status POST /__admin/tenants/:id/suspend POST /__admin/tenants/:id/activate UI is intentionally minimal — this is for the operator, not customers. Plain table, form, save button. Use the same CSS variables so admin doesn't fork the theme. Critically: the admin UI itself should NOT be tenant-scoped — it lives outside the multi-tenant world (on a dedicated host like admin.<our-domain>.com) and reads tenants directly via Mongoose. Commit: "phase-8: change 5 — tenant admin UI".

Prompt 8.7 — Change 6: Ops dashboard

Prompt 8.7 Build a read-only dashboard at /__admin/ops. Same role gate as 7.6. What it shows (per tenant, last 24h + last 7d): - Request count, error count. - Top 5 error paths. - Last successful payment, count of payments. - Last successful webhook event per provider. - For Square tenants: last successful inventory sync. - For tenants where webhooks have been quiet for > expected interval: red flag. Data source: a lightweight events log we add to in the existing middleware (don't introduce a new DB; reuse Mongo). Aggregate at request time — caching is a later optimization. The point of this dashboard is the 9am-Monday check. Optimize for "operator opens it, in 30 seconds knows whether anything needs attention". Commit: "phase-8: change 6 — ops dashboard".

Prompt 8.8 — Migrate the Phase-6 client

Prompt 8.8 THE critical migration. The phase-7 client is currently on its own deploy. We need to move them to the multi-tenant deploy without breaking them. Plan: 1. On the multi-tenant deploy, create the client as a Tenant document (using the admin UI from 7.6 or a script). Copy the siteConfigOverrides, hosts (including www. variant), preset. 2. Migrate the client's Mongo data: export from the single-tenant DB, import into the multi-tenant DB with the right tenantId. Use a migration script. Idempotent. Dry-run mode. 3. Stage: point a staging hostname at the multi-tenant deploy with the client's tenant. Compare side-by-side with the single-tenant deploy for parity. 4. Cutover: change DNS for the client's real host to the multi-tenant deploy. Watch the ops dashboard. Watch error rates. 5. After 24h of clean ops: decommission the single-tenant deploy (or keep it warm as rollback for another week — operator's call). Run a small real-world transaction post-cutover to confirm payments still flow. Commit migration scripts: "phase-8: migrate <client> to multi-tenant deploy".

Prompt 8.9 — Stress test the platform

Prompt 8.9 Stand up three more synthetic tenants (one per remaining vertical) on the multi-tenant deploy, each with a fake hostname (use /etc/hosts or a subdomain). Verify: - Each renders its own theme, content, payment provider. - A webhook for tenant A does not affect tenant B (forge events, observe). - The contrast validator runs across all four resolved theme bundles. - The ops dashboard shows all four in its tenant list with sane numbers. - A "suspend" on tenant A immediately makes A's hosts return the suspended page, while B/C/D keep serving. Report any anomaly. Anomalies = fixes BEFORE closing the phase. Commit fixes individually as "phase-8: stress test fix — <what>".

Prompt 8.10 — Close the phase + tag v1.0.0

Prompt 8.10 Final doc update pass. This is the v1.0.0 cut — the moment the template stops being "in development" and becomes "the platform". - CHANGELOG: a big Phase 8 / v1.0.0 entry. List all six changes, the migration of the phase-7 client, the stress-test outcomes. - package.json: bump to 1.0.0. - README.html (build-docs/): update doc-meta to v1.0.0. Add a short paragraph at the top noting "the template now serves multiple tenants from one deploy". - repo-context.html §05: add server/models/tenant.ts, server/middleware/resolve-tenant.ts, scripts/migrate/, admin/, ops/. - glossary.html: every term should now have a Phase-7-aware definition (tenant is now a real entity, not a defaulted string). - rules: review every cursor rule. Anything that said "tenantId defaults to 'default'" is now wrong by default — update. - briefs: any brief revision proposed in Phase 7 discoveries that survived review gets applied NOW, with a CHANGELOG note about which brief was edited and why. Tag v1.0.0. Fresh-clone test. Champagne.
05 — Acceptance criteria

Phase 8 is done when…

  • All six changes are implemented and committed individually.
  • The Phase-7 client is migrated and serving from the multi-tenant deploy with no customer-visible disruption.
  • Three synthetic tenants on the multi-tenant deploy demonstrate full isolation (data, theme, payments, webhooks).
  • The admin UI can create, edit, suspend tenants.
  • The ops dashboard shows per-tenant health.
  • Every cursor rule has been reviewed and updated for the multi-tenant world.
  • The briefs are consistent with the implemented system (revisions applied or proposals dismissed).
  • Tagged v1.0.0.

You now own a vertical-SaaS platform. Six discrete changes — not sixty — because Phase 1 invested in seams and Phase 5 invested in tenant-keyed resolution. The cost of the next client onboarding is now hours, not weeks. That is the thesis of the concept brief, paid out in code.

06 — What comes after v1.0.0

The build docs aren't done — they evolve

v1.0.0 means the template is feature-complete against the original three briefs. It does not mean the build docs are frozen. Per documentation-maintenance, the docs evolve as the platform evolves. Every new vertical, every new pack, every new shared concern gets its own phase doc.

What lives beyond v1.0.0

  • Phase 8+: Operational concerns that weren't on the v1.0 critical path — backups, observability deeper than the ops dashboard, rate limiting, abuse detection.
  • New verticals: A fifth vertical (when the operator has client momentum to justify it) gets its own preset folder, its own brand brief, its own onboarding phase prompt.
  • New feature packs: Packs are independently version-able. A "loyalty" pack or a "scheduling" pack ships as a Type-A addition with its own phase doc.
  • Brief revisions: The concept/template/visual-system briefs are living documents. Real client experience reshapes them on a roughly quarterly cadence.
The standing question

What's the next vertical, and which existing client conversation tells us it's ready? Every build phase from here forward starts by answering that.
