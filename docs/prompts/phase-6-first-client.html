PHASE 6 First Client · v0.6.0 → v0.7.0

Phase 6 — First client.

Drive one real client engagement through the template end-to-end. This is the "stage 1" moment from the concept brief — single tenant, deployed to its own infrastructure, real domain, real money, real customer. The template will lie to you about a half-dozen things; this phase is where you catch them.

01 — Prerequisites

Before you start this phase

  • Phase 5 closed, tagged v0.6.0. Visual system end-to-end.
  • You have a real client signed — not a hypothetical. The whole point of phase 6 is realness.
  • The client falls into one of the four verticals. (If they don't, defer them to a later phase; don't invent a fifth vertical mid-build.)
  • You have a content intake mechanism with the client — even if it's a shared doc. The template can't render what doesn't exist.

Phase 6 is a single-tenant deploy. We are NOT introducing multi-tenant routing yet. This client gets their own deploy of the template (forked or branched), with tenantId='default' on its own DB. Phase 7 is when we collapse multiple deploys into one.

Discoveries become brief revisions, not template hacks. When the client wants something the template can't do cleanly, the first response is "is this a brief problem or a template problem?" — not "let me add a special-case branch".

03 — Change taxonomy in this phase

What kind of changes are we making?

Type A
Client tenant files

Creating theme/tenants/<client>.pick.ts and seed content is additive. No existing code path changes. This is the right kind of change for a client-specific phase.

Type B
(careful here)

If the client needs a token that doesn't exist, the brief discipline kicks in. Either escalate to vertical-tokens (changes everyone), or add a leaf to ThemeTokens (Type B — ripples to every layer). Do not silently inline.

Type C
Discovered structural needs

If the client surfaces a need for a structural template change (e.g. orders need a "scheduled pickup date"), it's Type C. Document, propose, decide whether to do it now (delays launch) or after (carry as debt with explicit deadline).

04 — Prompts to run, in order

The phase-6 prompt sequence

Phase 6 mixes prompts with operator-only work (talking to the client, picking a domain, etc). Prompts are flagged CLAUDE or OPERATOR.

Prompt 6.1 OPERATOR — Sign the client + write a project brief

Prompt 6.1 — operator task Before any prompt to Claude, you (the operator) must: 1. Have the client signed (or have explicit OK to use them as the phase-6 test). 2. Write a one-page project brief: client name, vertical, what they sell, who their customers are, what content they have ready, what payment provider they want, what domain, what launch date. 3. Confirm the client's vertical IS one of the four. If not, STOP — do not invent a fifth vertical here. Save the brief at build-docs/clients/<client>.brief.md. Every following prompt assumes it exists.

Prompt 6.2 CLAUDE — Stand up the client's instance

Prompt 6.2 We have a real client at build-docs/clients/<client>.brief.md. Read it. This is a single-tenant deploy. Steps: 1. Confirm the client's vertical key matches a registered preset. 2. Decide infrastructure path: separate deploy of the template (current state), OR add the client as a tenant in a future shared deploy (NOT YET — that's phase 7). For phase 6, separate deploy. 3. Choose a tenant id (kebab-case, no reserved words). 4. Run init:vertical -- --preset=<v> --tenant=<client-tenant-id> against a NEW database (not the dev one). 5. Walk the seeded app for that tenant. Note any seed content that needs replacing with the client's actual content. Report what was created and what content is still placeholder.

Prompt 6.3 OPERATOR — Content intake

Prompt 6.3 — operator task Get real content from the client. Bare minimum: - Real product/menu list with real prices. - Real "About" copy (1-3 paragraphs in their voice). - Real photos (or commit to using stock as a stopgap, with replacement timeline). - Real contact info, hours, location. - Logo file (vector if possible). Put it all in a structured form Claude can read: a single content.intake.md or a folder under build-docs/clients/<client>/content/. Until this is done, no further Claude prompts. The template can't render fake content into a real launch.

Prompt 6.4 CLAUDE — Load the real content

Prompt 6.4 We now have real content at build-docs/clients/<client>/content/. 1. Replace the seeded Products in the client's tenant with the real product list. 2. Update the Page records (Home, About, etc) with real copy. 3. Upload logo + photos to whatever asset host we're using (per stack-context). 4. Update the FormDefinition for "Contact" or equivalent. For each replacement, log what was placeholder before and what is real now. Do NOT change any code. Only data. If any required content is missing, list it (don't hallucinate replacements). Commit: "phase-6: real content for <client>".

Prompt 6.5 CLAUDE — Real brand picks

Prompt 6.5 The client has visual preferences. Create theme/tenants/<client>.pick.ts based on those. Process: 1. Read theme/verticals/<v>.tokens.ts to see what's vertical-default for this client. 2. List what the client wants different (from the brief or from a call). 3. For each difference: is it expressible as a curated pick (a named option we offer) or does it need a leaf override? - If pick: add it to the pick file. - If override: add it to <client>.override.ts, BUT only if it's a single leaf. If it's structural, FLAG IT and we'll decide whether to escalate or refuse. 4. Run the contrast validator against the resolved tokens. If it fails, the client's preference is breaking accessibility — surface that with options. 5. Render the site and screenshot the home page. Save the screenshot to build-docs/clients/<client>/screenshots/. Commit: "phase-6: brand picks + (maybe) leaf overrides for <client>".

Prompt 6.6 CLAUDE — Pre-flight checks

Prompt 6.6 Before any deploy, run a pre-flight pass against the client's tenant: 1. doctor.ts — must pass with the client's env. 2. contract:check — must pass. 3. contrast validator on resolved tokens for <client> — must pass. 4. Walk every page in the rendered app, log any 404, 500, broken image, or "lorem". 5. Confirm payment provider is configured (test mode is fine right now; live mode toggle in 6.8). 6. Confirm Auth0 (if auth pack is on) has the right callback URLs configured for the client's domain. Produce a pre-flight report. Do not proceed if any check fails.

Prompt 6.7 OPERATOR — Deploy to staging

Prompt 6.7 — operator task Deploy the client's instance to a staging URL (e.g. clientname.staging.<your-domain>.com). SSL configured. Walk the client through it. Capture their feedback in build-docs/clients/<client>/feedback-staging.md. Iterate via additional content prompts or pick adjustments (NOT code changes — code changes mean a template-level discovery; route those through 6.10).

Prompt 6.8 OPERATOR — Production cutover

Prompt 6.8 — operator task Production cutover. Steps: 1. Point the client's real domain at production infra. 2. Issue SSL certificate. 3. Switch payment provider from test to live mode. Update STRIPE_SECRET_KEY / SQUARE_ACCESS_TOKEN, etc. 4. Update Stripe / Square webhook URLs to the live domain. 5. Run doctor in production mode — confirm webhook signatures verify against live keys. 6. Smoke-test: make a real $1 purchase. Refund it from the provider dashboard. Confirm order flips to refunded in our DB. Commit any env or runbook changes to build-docs/clients/<client>.md. Tag the production deploy with the template version: v0.7.0-client-<name>.

Prompt 6.9 CLAUDE — Write the runbook

Prompt 6.9 Create build-docs/clients/<client>.md as a runbook. Sections: - Tenant info (id, vertical, deploy URL, infra notes). - Payment provider details (which one, account id, where to refund, where to view payouts). - Content edit workflow (where pages live in the admin, who has access). - Common operations: add a product, change hours, post an update, replace a photo. - Troubleshooting: known false alarms, where logs are, how to escalate. - Contact: who at the client to call, who internally owns this account. This is the doc the operator (or a teammate) will open at 9am Monday when the client emails about a typo. Make it scannable.

Prompt 6.10 CLAUDE — Discoveries log + brief revision proposals

Prompt 6.10 Throughout phases 6.1-6.9 you (and the operator) have noticed things. Some are minor. Some contradict the briefs. Write them up. Create build-docs/discoveries/<client>-phase-6.md with sections: ## What worked as the briefs predicted ## What was easier than expected ## What was harder than expected ## What surprised us ## What contradicted a brief ## Recommended brief revisions (one per contradiction) For each "recommended brief revision" propose: - Which brief (concept / template / visual-system). - What sentence/section changes. - Why this change does not weaken the brief's thesis. Do NOT change any brief yet. Proposals are reviewed by the operator before any brief is edited.

Prompt 6.11 CLAUDE — Close the phase

Prompt 6.11 Doc update pass: - Append Phase 6 entry to CHANGELOG. Include: client name (or codename if not public), vertical, what brand picks were made, what discoveries surfaced. - Bump package.json to 0.7.0. - Update repo-context.html §05 to show theme/tenants/<client>.*.ts and build-docs/clients/<client>.*. - Update README.html doc-meta version. - List OPEN_QUESTIONS for Phase 7 — especially: any place where the single-tenant deploy revealed assumptions the multi-tenant model will have to address. Tag v0.7.0. The fresh-clone test now needs to also verify that an init:vertical for the client's vertical still produces a coherent demo (not the client's real data — never seed production data into the template).
05 — Acceptance criteria

Phase 6 is done when…

  • The client's site is live at its real domain, with SSL, in payment provider live mode.
  • At least one real customer transaction has gone through end-to-end (operator and Claude both verify in their respective dashboards).
  • theme/tenants/<client>.pick.ts exists with real picks; .override.ts exists (possibly empty, possibly one leaf).
  • The runbook at build-docs/clients/<client>.md is written and scannable.
  • The discoveries log exists with at least one entry per section.
  • Brief revisions, if any, are proposed (not yet applied) in the discoveries log.
  • Tagged v0.7.0 and v0.7.0-client-<name>.

This is the proof. Concept + template + visual system have produced a real working site for a real paying customer. Phase 7 collapses the single-tenant deploy into a multi-tenant one — but that's a refactor with a known target. The hard part is over.

06 — Status & handoff to Phase 7

What's true at the end of Phase 6

One real client live, single-tenant. The template knows how to take itself the last mile from "demo" to "production". Phase 7 is the architectural step everyone was supposed to be afraid of, which we de-risked starting at Phase 1.

What Phase 7 will pick up

  • The six discrete multi-tenant changes from the template architecture brief: Tenant model, host middleware, getSiteConfig swap, SSL strategy, client config admin, ops dashboard.
  • Migrating the phase-6 client from "their own deploy" to "shared multi-tenant deploy" — same DB rows, just routed via host.
Open question to carry forward

Did the phase-6 client need anything that tenantId on every model doesn't already buy us? Phase 7's first prompt audits this — if there's a surprise here, better to find it before flipping the host-based routing on.
