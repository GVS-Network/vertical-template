PHASE 0 Bootstrap · Template v0.0.0 → v0.1.0

Phase 0 — Bootstrap.

Fork baseapp into vertical-template, install the doc family and the cursor rules, verify the dev loop boots end-to-end. Nothing about verticals or SiteConfig yet — just a running MERN that we own and that knows how to talk to Claude.

01 — Prerequisites

Before you start this phase

  • You have read the three reference briefs (concept, template, visual system) at least once.
  • You have read all four context docs: repo-context, stack-context, glossary, session-starter.
  • You have access to the baseapp repo (Anthropic's frozen MERN starter we forked from) and to askanddeliverwebapp for reference patterns.
  • You have Node 20 and a clean shell. No global Mongo or Stripe CLI required yet.

One commit per prompt. Phase 0 is the easiest place to install the habit. If you do nine commits in this phase, you should be able to look at the log and reconstruct which prompt produced which change. If you can't, the discipline is already drifting.

03 — Prompts to run, in order

The phase-0 prompt sequence

Open a fresh Cursor session. Paste the session starter first (with PHASE=0). Then run these prompts one at a time. Wait for Claude to finish each before sending the next. Commit between each.

Prompt 0.1 — Repo fork & inventory

Prompt 0.1 We are at the start of Phase 0 (Bootstrap) of the vertical-template build. Before any code changes, do an inventory pass: 1. Read every file in the repo root (package.json, tsconfig*, .env.example, README, etc). 2. Read the top-level structure of /server, /client, and any other top-level folders. 3. Read the existing baseapp README in full if one exists. Then produce a single markdown report in build-docs/inventory.md with: - What stack/tools baseapp is actually running (versions from package.json, not from memory). - Every top-level folder, one-line purpose. - Anything that contradicts what stack-context.html claims (we want to learn from the diff, not paper over it). - Anything baseapp already has that the template will need to keep (auth scaffolding, error middleware, etc). - Anything baseapp has that the template will need to remove or replace (placeholder routes, demo content, hardcoded styling). Do not change any code yet. This is a read-only pass.

Prompt 0.2 — Reconcile stack-context with reality

Prompt 0.2 Based on inventory.md, propose edits to build-docs/contexts/stack-context.html so it matches what is actually in the repo right now, not what we thought we'd inherit. Specifically: - If a dependency version differs, update stack-context. - If baseapp uses a lib we marked "forbidden", flag it; do not silently bless it. List it in a "to remove in phase 0" callout we will resolve before phase end. - If stack-context references something baseapp doesn't have yet, leave it; we'll add it when its phase comes. Output: a diff for stack-context.html plus a short "what changed and why" note I will paste into the CHANGELOG.

Prompt 0.3 — Install the cursor rules

Prompt 0.3 Copy the cursor rule files I have in build-docs/.cursor/rules/ and build-docs/.cursorrules into the actual cursor-active locations: - /.cursorrules (repo root, legacy format) - /.cursor/rules/*.mdc (modern globbed format) After copying, verify: - Every .mdc file has valid frontmatter (description, globs OR alwaysApply). - No two .mdc files claim the same globs without a clear reason. - .gitignore does NOT exclude .cursor/ — these rules are part of the project. Then commit with message: "phase-0: install cursor rules (legacy + modern)".

Prompt 0.4 — Install build-docs and reference briefs

Prompt 0.4 Copy the entire build-docs/ folder I have prepared into the repo root. Then copy the three reference briefs (platform-concept-brief.html, template-architecture-brief.html, visual-system-brief.html) into build-docs/briefs/. After copying: - Open build-docs/README.html and verify every relative link in §02 (the doc family index) resolves to an actual file. - If any link is broken, fix the path; do not delete the card. - Make sure build-docs/assets/doc-system.css is one file, referenced consistently from every HTML. Commit: "phase-0: install build-docs and reference briefs".

Prompt 0.5 — CHANGELOG and version files

Prompt 0.5 Create a root-level CHANGELOG.md following Keep-a-Changelog format. Seed it with one entry: ## [0.1.0] — <today's date> ### Added - Forked baseapp into vertical-template. - Installed build-docs/ family (README, contexts, rules, phase prompts). - Installed cursor rules (.cursorrules and .cursor/rules/*.mdc). - Stack-context reconciled against actual repo state. Also bump package.json version to 0.1.0 (it is currently whatever baseapp left it at). Commit: "phase-0: cut v0.1.0".

Prompt 0.6 — Create scripts/doctor.ts

Prompt 0.6 Create scripts/doctor.ts. Its job is to fail loudly if the local dev environment is wrong. For phase 0 it must check: 1. Node version is >= 20 (read process.versions.node). 2. All env vars listed in .env.example are present in .env (read both files, compare keys). 3. MONGODB_URI connects (attempt a Mongoose connect with a 3s timeout). It must NOT check feature-pack-specific or vertical-specific things yet — those get added by their own phases. Wire it into package.json as "doctor": "tsx scripts/doctor.ts" and add a note in README.html (the build-docs one) under §03 about running it. Commit: "phase-0: scripts/doctor.ts (env+mongo check)".

Prompt 0.7 — Verify dev loop end-to-end

Prompt 0.7 Walk me through verifying the dev loop boots cleanly. Specifically: 1. Run npm install. Report any peer-dep warnings or version conflicts. 2. Run npm run doctor. It must pass. 3. Run npm run dev. Server must come up on its expected port; client must come up on its expected port. 4. Hit the client URL in a browser. The baseapp default page must render. 5. Hit one server health endpoint (if baseapp ships one) or create a minimal GET /api/health that returns { ok: true } if none exists. If anything is broken, FIX the smallest thing that unblocks the loop. Do NOT start refactoring baseapp's structure — that is later phases' work. Phase 0 is "it runs". After all five checks pass, commit any fixes: "phase-0: dev loop green".

Prompt 0.8 — Close the phase

Prompt 0.8 We are closing Phase 0. Do the doc-update pass per build-docs/rules/documentation-maintenance.html: 1. Append a "Phase 0 — closed" entry to CHANGELOG.md listing every prompt run and its outcome. 2. Update build-docs/README.html doc-meta version to v0.1.0 if not already there. 3. List, in a short note for me, any open questions or surprises from this phase that should inform phase 1 (these go into the next session-starter's OPEN_QUESTIONS slot). 4. Confirm: no Type-A, Type-B, or Type-C change taxonomy work was done this phase (that starts in phase 1). If any happened accidentally, flag it. Commit: "phase-0: close v0.1.0" and tag the commit v0.1.0.
04 — Acceptance criteria

Phase 0 is done when…

  • npm install && npm run doctor && npm run dev works from a fresh clone.
  • build-docs/ exists at repo root with every file from the prepared family.
  • .cursorrules exists at repo root and .cursor/rules/ contains all .mdc files.
  • CHANGELOG.md exists with a v0.1.0 entry that accurately reflects what shipped.
  • package.json version is 0.1.0.
  • The commit v0.1.0 is tagged.
  • You can hand this repo to someone else, point them at build-docs/README.html, and they could get oriented without asking you anything.

The "fresh clone" test. Before tagging v0.1.0, clone the repo into a different directory, run the three commands above, and confirm it works. If it doesn't, something in your local dev env is masking a missing file. Fix that before tagging.

05 — Status & handoff to Phase 1

What's true at the end of Phase 0

The template is a forked, renamed, version-tagged copy of baseapp with the doc family and cursor rules installed and a verified dev loop. No template-specific logic exists yet. It is still functionally baseapp.

What Phase 1 will pick up

  • The SiteConfig type, the getSiteConfig(req) seam, the dual-file contract discipline.
  • Threading tenantId through every Mongoose model (defaulted to 'default' until phase 7).
  • The first real Type-A vs Type-B vs Type-C decisions.
Open question to carry forward

Did anything in baseapp surprise us that the template should explicitly preserve or explicitly remove? Phase 1's first prompt should resolve this before touching SiteConfig.
