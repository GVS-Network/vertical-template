PHASE 1 SiteConfig & Seams · v0.1.0 → v0.2.0

Phase 1 — SiteConfig & seams.

Establish the one TypeScript shape every vertical and every client will fill in differently — and the two seams (getSiteConfig, getPaymentProvider) that everything else hangs off of. Thread tenantId through every model so the phase-7 multi-tenant jump is a six-change swap, not a rewrite.

01 — Prerequisites

Before you start this phase

  • Phase 0 is closed and tagged v0.1.0. npm run doctor and npm run dev both pass.
  • You have re-read the template architecture brief, especially the SiteConfig section and the six-change phase-2 jump (which we now call phase-7 to match this build doc's numbering).
  • You have re-read contexts/glossary.html — specifically the Type A / Type B / Type C taxonomy.

The dual-file contract starts now. Every time SiteConfig or ThemeTokens changes, two files must be touched in the same commit: the type and a default. Cursor rule 50-siteconfig-contract.mdc enforces this; do not silently override it.

02 — What this phase produces

Deliverables

types/site-config.tsthe one shape

The canonical SiteConfig type: vertical, branding placeholder, feature flags (all five packs stubbed off), payment provider hint, contact info, locale. Plus ThemeTokens sibling for the visual system layer.

types/site-config.defaults.tsthe dual-file partner

A defaultSiteConfig object that satisfies the type. Any new required field added to the type must get a default here in the same commit — that is the contract.

server/seams/get-site-config.tsseam #1

getSiteConfig(req). In phase 1 it returns defaultSiteConfig every time. In phase 7 it switches to host-based lookup. Nothing else changes about its callers.

server/seams/get-payment-provider.tsseam #2 (stub)

getPaymentProvider(siteConfig). In phase 1 it throws NotImplementedYet. Phase 3 fills it in. The seam exists so callers can be written today.

tenantId on every modelforward investment

Every Mongoose schema gets a tenantId: string field, indexed, defaulted to 'default'. Every query goes through a helper that scopes by tenantId. This is the single largest investment in phase 7 being cheap.

scripts/contract-check.tsenforced by CI

A script that asserts: every field in SiteConfig has a corresponding entry in defaultSiteConfig. Runs in CI; fails the build if the contract breaks.

03 — Change taxonomy in this phase

What kind of changes are we making?

Per the template architecture brief, every template change is classified A, B, or C. Phase 1 is dominated by Type-A and one careful Type-C.

Type A
Adding fields to SiteConfig

Adding vertical, contact, features, locale to the type. Pure additive, defaulted, no caller has to change. Most of phase 1 is this.

Type B
(deferred to phase 5)

Visual tokens get added in phase 5. We declare ThemeTokens as an empty interface here but do not propagate to overrides yet.

Type C
Threading tenantId

Touching every model, every query, every controller is a Type-C change — it ripples. We do it once, here, deliberately, so phase 7 doesn't have to.

04 — Prompts to run, in order

The phase-1 prompt sequence

Prompt 1.1 — Carry forward open questions

Prompt 1.1 Open the OPEN_QUESTIONS from Phase 0's close (in CHANGELOG.md or in the previous session's notes). For each open question, decide one of three things: 1. Resolve now — answer it before we touch SiteConfig. 2. Defer with a written rationale — note which later phase will pick it up. 3. Drop — explain why it's no longer relevant. Do not start any Phase 1 work until every open question has one of these three outcomes.

Prompt 1.2 — Draft the SiteConfig type

Prompt 1.2 Create types/site-config.ts. Based on template-architecture-brief.html, the shape should include AT MINIMUM: - vertical: 'bar-restaurant' | 'food-truck' | 'farm-source' | 'screen-printer' | 'generic' - tenantId: string (defaulted 'default' until phase 7) - features: { catalog: boolean; content: boolean; intake: boolean; payments: boolean; auth: boolean } - payment: { provider: 'stripe' | 'square' | 'none' } - branding: { name: string; tagline?: string } - contact: { email?: string; phone?: string; address?: string } - locale: { currency: string; timezone: string; lang: string } Also create types/theme-tokens.ts with an EMPTY interface ThemeTokens — phase 5 fills it. Do NOT add any field that doesn't appear in the brief or in this prompt. If you think one is missing, FLAG IT for me, don't sneak it in.

Prompt 1.3 — Defaults file (the dual-file partner)

Prompt 1.3 Create types/site-config.defaults.ts exporting defaultSiteConfig: SiteConfig with sensible neutral values: - vertical: 'generic' - tenantId: 'default' - features: all five FALSE (we toggle them on per-vertical in phase 4) - payment.provider: 'none' - branding: { name: 'Untitled Site' } - contact: { } (all optional) - locale: { currency: 'USD', timezone: 'America/Chicago', lang: 'en-US' } This file MUST satisfy the type as written. If TS complains, the type is the source of truth — adjust defaults, not the type, unless we discussed it. Commit both files together: "phase-1: SiteConfig type + defaults (dual-file contract established)".

Prompt 1.4 — Build the contract check

Prompt 1.4 Create scripts/contract-check.ts. It must: 1. Read types/site-config.ts and extract every top-level field of SiteConfig (including nested required fields). 2. Read types/site-config.defaults.ts and confirm defaultSiteConfig has every required field. 3. Exit non-zero if any required field is missing a default. Wire it into: - package.json scripts: "contract:check": "tsx scripts/contract-check.ts" - package.json prebuild hook so CI catches contract drift before deploy - scripts/doctor.ts so local dev also catches it Then deliberately break the contract (add a required field with no default), run the check, confirm it fails. Then fix it. Commit: "phase-1: contract-check script + doctor wiring".

Prompt 1.5 — getSiteConfig seam

Prompt 1.5 Create server/seams/get-site-config.ts: export function getSiteConfig(req: Request): SiteConfig { return defaultSiteConfig; } That is the ENTIRE implementation for phase 1. The signature accepting req is the whole point — phase 7 changes the body to read req.hostname and look up a Tenant document. No caller changes. Wire it into one place: a middleware that attaches req.siteConfig = getSiteConfig(req). Add the type augmentation to a types/express.d.ts file. Find one existing route in baseapp (any one) and refactor it to read req.siteConfig.branding.name instead of a hardcoded string, just to prove the seam flows end-to-end. Commit: "phase-1: getSiteConfig seam + req.siteConfig middleware".

Prompt 1.6 — getPaymentProvider stub seam

Prompt 1.6 Create server/seams/get-payment-provider.ts: export function getPaymentProvider(siteConfig: SiteConfig): PaymentProvider { throw new Error('NotImplementedYet: payments arrive in phase 3'); } Define a PaymentProvider interface (charge, refund, listProducts, syncInventory) but leave it as a type-only declaration. No implementations yet. This seam exists so phase-2 feature-pack scaffolding can be written against a stable signature. Commit: "phase-1: getPaymentProvider stub seam + PaymentProvider interface".

Prompt 1.7 — Thread tenantId through every model

Prompt 1.7 This is the Type-C change of Phase 1. Do it carefully. 1. Find every Mongoose schema in the repo. 2. For each, add tenantId: { type: String, required: true, default: 'default', index: true }. 3. Add a compound index of (tenantId, <the primary lookup field>) where it makes sense. 4. Create server/db/scoped.ts exporting a helper scoped(Model, req) that returns Model with a tenantId filter pre-baked. Use it everywhere instead of raw Model.find / Model.findOne. 5. Update every controller/service that queries a model to use the helper. 6. Run all tests (if any). Fix what breaks. Produce a list of every file you touched and why. Commit as ONE commit: "phase-1: thread tenantId through all models (Type-C, defaulted 'default')". If you find a model you can't safely scope (e.g. system-wide config), call it out — do not silently exclude it.

Prompt 1.8 — Update docs

Prompt 1.8 Per build-docs/rules/documentation-maintenance.html doc update matrix, this phase has touched: - Type contract → update glossary if any term shifted - Seams → ensure repo-context.html §05 file map shows server/seams/ - tenantId rule → already in cursor rule 70-multi-tenant-seams.mdc; verify it still reads correctly given what we actually built Append a Phase 1 entry to CHANGELOG.md. Bump package.json to 0.2.0. List any new OPEN_QUESTIONS for Phase 2 — especially any place where a feature pack will have to interact with tenantId in a way we didn't fully figure out yet. Commit: "phase-1: close v0.2.0". Tag v0.2.0.
05 — Acceptance criteria

Phase 1 is done when…

  • types/site-config.ts and types/site-config.defaults.ts both exist; contract:check passes.
  • req.siteConfig is populated on every request and at least one route reads from it.
  • getPaymentProvider exists as a throwing stub; nothing calls it yet.
  • Every Mongoose schema has tenantId; every query goes through scoped().
  • npm run doctor includes the contract check and still passes.
  • The fresh-clone test from phase 0 still works.
  • The commit v0.2.0 is tagged.

The shape of phase 1 sets the shape of every later phase. If SiteConfig is sloppy here, every vertical preset will paper over it with hacks, and the phase-7 multi-tenant jump will not be six changes — it'll be sixty. Spend the extra hour now.

06 — Status & handoff to Phase 2

What's true at the end of Phase 1

There is one shape, two seams, and one defaulted tenant. The template is now a MERN that knows it is going to become many sites — even though right now it is only one. Phase 2 stops talking about shape and starts building behavior.

What Phase 2 will pick up

  • The five feature packs (catalog, content, intake, payments, auth), each scaffolded as a self-contained module.
  • Toggle-mounting: feature packs read req.siteConfig.features.* and silently skip when off.
  • Each pack ships a schema, a service, a controller, a router, and a client component family.
Open question to carry forward

Should feature packs read SiteConfig directly, or should the server middleware filter them out before the request even reaches the pack's router? Phase 2's first prompt resolves this.
