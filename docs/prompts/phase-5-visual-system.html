PHASE 5 Visual System · v0.5.0 → v0.6.0

Phase 5 — Visual system.

Implement the four-layer token resolution from the visual-system brief: foundation → vertical → tenant pick → tenant override. Build resolveTokens(), the contrast validator, and CSS variable emission. Enforce leaf-only overrides. Phase 5 turns one React app into many visually distinct sites.

01 — Prerequisites

Before you start this phase

  • Phase 4 closed, tagged v0.5.0. All four presets stamp cleanly.
  • You have re-read the visual system brief in full. The four-layer model and leaf-only rule should be top of mind.
  • The Phase-4 open question is resolved: opinionated brand defaults per vertical, or neutral. Phase 5's prompts assume opinionated (per concept brief) unless overridden.

The brief is the spec. Don't reinvent the resolution model in code. If you find yourself wanting a fifth layer, or a "smart" merge, stop — that's a brief revision, not a code change. Either escalate or stay strict.

02 — What this phase produces

Deliverables

ThemeTokens (filled in)the leaves

types/theme-tokens.ts stops being empty. Concrete leaf tokens: color.bg, color.ink, color.accent, type.heading.family, type.body.family, radius.md, shadow.sm, etc. Every token is a primitive (string/number).

Four foundation layersfiles, not magic

theme/foundation.tokens.ts (neutral defaults), theme/verticals/<v>.tokens.ts (one per preset), theme/tenants/<t>.pick.ts (the curated pick list), theme/tenants/<t>.override.ts (rare, leaf-only).

resolveTokens()the merge function

Pure function: (siteConfig, tenantId) → ResolvedTokens. Reads the four layers in order, deep-merges, enforces leaf-only rule on the override layer.

Contrast validatoraccessibility guardrail

For every text-on-bg pair declared in tokens, compute WCAG contrast ratio. Fail CI (and doctor) if below 4.5:1 for body text. Warn under 7:1 (AAA).

CSS variable emissionhow tokens reach the DOM

Server-side: tokens are resolved per request and emitted as a <style>:root{--…}</style> in the HTML head. Client React reads var(--color-accent) in styles. No JS state for tokens.

Per-vertical opinionated brandcredibility on day one

Each theme/verticals/*.tokens.ts is a real, considered brand expression — not a placeholder. A food-truck preset boots looking like a food truck, not like generic SaaS.

03 — The four-layer model (recap)

Resolution order, leaf-only at the bottom

How a token gets its final value

The visual-system brief defines four layers, applied in this order: foundation (neutral defaults that exist for every token), then vertical (the preset's opinionated overrides — typography, palette, voice), then tenant pick (the client chooses from a curated set of options — like picking from a swatch book), then tenant override (the rare leaf-level escape hatch when a client needs one thing changed exactly). Each layer can only set leaves. No layer can rewrite the structure of the tree.

1foundation.tokens.ts
Every leaf defaulted. The contract: if every other file disappeared, this would still render a valid site.
2verticals/<v>.tokens.ts
Opinionated per vertical. Overrides foundation leaves. Cannot introduce new leaves the type doesn't know about.
3tenants/<t>.pick.ts
Client picks from curated options. Each pick is a named bundle (e.g. palette: 'forest') which expands to leaf changes.
4tenants/<t>.override.ts
Rare. Leaf-only. Validator enforces. Every override here is a story we have to tell.

Minimum leaves (every value a primitive): color.bg, color.bg.panel, color.ink, color.ink.soft, color.ink.muted, color.rule, color.accent, color.accent.soft, color.green, color.amber, color.blue, color.danger type.heading.family, type.heading.weight, type.heading.tracking, type.body.family, type.body.weight, type.mono.family size.base, size.h1, size.h2, size.h3, size.body, size.small, size.mono space.xs, space.sm, space.md, space.lg, space.xl radius.sm, radius.md, radius.lg shadow.sm, shadow.md motion.duration.fast, motion.duration.med, motion.ease Update types/theme-tokens.ts. EVERY leaf MUST appear in foundation.tokens.ts in the next prompt. The dual-file contract from Phase 1 now also covers theme. Commit: "phase-5: ThemeTokens type (full leaf set)".

Prompt 5.3 — foundation.tokens.ts

Prompt 5.3 Create theme/foundation.tokens.ts exporting const foundation: ThemeTokens with neutral, accessible, sensible defaults. Goal: a site rendered with ONLY foundation tokens should look quiet, legible, and unobjectionable. Like a well-typeset blank page. Reference points (not prescriptions): - color.bg: a neutral off-white or cream - color.ink: dark, but not pure #000 (1a1a1a-ish) - color.accent: ONE accent that passes 4.5:1 on bg - type.heading.family: a serif by default (Fraunces, Georgia stack) - type.body.family: a humanist sans (Inter Tight, system stack) Update scripts/contract-check.ts to assert: every leaf in ThemeTokens has a value in foundation.tokens.ts (same dual-file rule as SiteConfig). Commit: "phase-5: foundation tokens + contract check".

Prompt 5.4 — Per-vertical token files

Prompt 5.4 For each vertical, create theme/verticals/<v>.tokens.ts exporting const verticalTokens: DeepPartial<ThemeTokens>. Build them ONE AT A TIME, in the order the brand briefs were strongest. After each, hot-reload and look at the rendered demo tenant for that vertical. The tokens should be opinionated: - bar-restaurant: warm, low-light, ink-on-cream, serif headings. - food-truck: high-contrast, punchy accent, condensed sans headings. - farm-source: green-leaning, earthy, paper-bag textures via accent-soft. - screen-printer: print-shop bold, almost monospace headlines, ink black. DO NOT introduce new leaves. If you find one you need, STOP, update ThemeTokens, update foundation, then continue — never silently extend. Commit per vertical: "phase-5: theme/verticals/<v>.tokens.ts".

Prompt 5.5 — Tenant pick + override files (stubs)

Prompt 5.5 For the demo tenants seeded in Phase 4, create: theme/tenants/demo-<v>.pick.ts — empty for now (no picks), exporting const pick: DeepPartial<ThemeTokens> = {}. theme/tenants/demo-<v>.override.ts — empty for now, exporting const override: DeepPartial<ThemeTokens> = {}. The point is to prove the file conventions and the registry. Real tenants in Phase 6 will populate these. Add theme/tenants/registry.ts that imports all of them keyed by tenantId. Commit: "phase-5: tenant pick + override file stubs".

Prompt 5.6 — resolveTokens()

Prompt 5.6 Create theme/resolve.ts exporting: export function resolveTokens(siteConfig: SiteConfig, tenantId: string): ThemeTokens Steps: 1. Start with foundation. 2. Deep-merge verticals[siteConfig.vertical] on top. 3. Deep-merge tenants[tenantId].pick on top. 4. Deep-merge tenants[tenantId].override on top, BUT FIRST validate it: - Every key path in override must terminate at a leaf in ThemeTokens. - Any non-leaf path → throw with a clear error pointing at the offending file. 5. Return the resolved tokens. Write unit tests for: - foundation-only resolves correctly. - vertical overrides foundation leaf-by-leaf. - pick adds bundled leaf changes. - override at non-leaf throws. - override at leaf wins. Commit: "phase-5: resolveTokens + leaf-only validator + tests".

Prompt 5.7 — Contrast validator

Prompt 5.7 Create theme/validate-contrast.ts. Given ResolvedTokens, compute WCAG contrast for these pairs: - color.ink on color.bg - color.ink.soft on color.bg - color.bg on color.accent (button text scenario) - color.ink on color.bg.panel Rules: - body text pairs (ink/bg, ink/panel) must be >= 4.5:1 — fail otherwise. - accent pairs must be >= 3.0:1 (large text) — warn under 4.5:1, fail under 3.0:1. Wire it into: - scripts/doctor.ts: validate resolved tokens for every (vertical × demo-tenant) combo. - resolveTokens()'s test suite. - CI prebuild. Commit: "phase-5: contrast validator + CI wiring".

Prompt 5.8 — CSS variable emission

Prompt 5.8 Create theme/emit-css-vars.ts: export function emitCssVars(tokens: ThemeTokens): string Outputs a string like: :root { --color-bg: #f5f1ea; --color-ink: #1a1a1a; --color-accent: #b8451f; /* ... etc ... */ } Wire it into the server's HTML response: - On each request, after resolveTokens(siteConfig, tenantId), call emitCssVars and inject into the <head> as a <style> element BEFORE any other styles. - The client's components reference var(--…) only. No JS reading tokens at runtime. Convert at least one existing pack's component (pick something visible like ProductCard) to use the CSS variables instead of hardcoded values. Walk a demo tenant for each vertical and confirm visual differentiation. Commit: "phase-5: CSS var emission + first component converted".

Prompt 5.9 — Convert remaining components

Prompt 5.9 Audit every component across all five packs (client/features/<pack>/components/) and convert hardcoded colors, fonts, spacings to var(--…) references. Rules: - One commit per pack. "phase-5: convert client/features/<pack> to CSS vars". - If a component uses a value not in ThemeTokens, that's a signal — either it's a one-off legitimately, or a leaf is missing. Decide deliberately each time. - Visual regression: take a screenshot of each main page per vertical before and after; eyeball. End-of-prompt report: which components were converted, which were left as-is and why.

Prompt 5.10 — Close the phase

Prompt 5.10 Doc update pass: - Append Phase 5 entry to CHANGELOG. - Bump package.json to 0.6.0. - Update repo-context.html §05 file map to include theme/ tree. - Update glossary.html: every term from the visual-system brief — token, leaf, pick, override, resolution, foundation, vertical layer — should have a concise entry. - Update README.html doc-meta version. - List OPEN_QUESTIONS for Phase 6 — especially: what's the workflow when a client wants something the pick list doesn't offer? 05 — Acceptance criteria

Phase 5 is done when…

  • ThemeTokens is fully populated; contract:check covers it.
  • foundation.tokens.ts + four vertical token files all exist.
  • resolveTokens() exists, has tests, and rejects non-leaf overrides.
  • The contrast validator fails CI on bad palettes and is wired into doctor.
  • Booting the same React app with different BOUND_TENANT_IDs renders four visually distinct sites.
  • Components reference var(--…) rather than hardcoded values.
  • Tagged v0.6.0.

This is the demo moment. By the end of phase 5, you can sit a prospective client in front of a laptop and show them: same template, four wildly different sites, instantly. That's the pitch the concept brief promised.

06 — Status & handoff to Phase 6

What's true at the end of Phase 5

Four-layer token resolution works. Contrast is enforced. Every component flows from tokens. The template is fully themeable, but no real client has shaped it yet. Phase 6 brings a real engagement through the template end-to-end.

What Phase 6 will pick up

  • Driving an actual client build through the template — the concept-brief's "stage 1" moment.
  • Populating one real tenants/<name>.pick.ts and possibly .override.ts.
  • Discovering everything the template assumes that turns out not to be true.
Open question to carry forward

When a client request can't be satisfied by foundation+vertical+pick+leaf-override — i.e. it wants a structural change — is the answer (a) escalate to the vertical layer (changes everyone in that vertical), (b) introduce a new pick option, or (c) say "no, that's not what this template does"? Phase 6 will surface this; we need the policy ready.
