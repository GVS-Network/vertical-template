PHASE 4 Verticals · v0.4.0 → v0.5.0

Phase 4 — Verticals.

Four presets: bar-restaurant, food-truck, farm-source, screen-printer. Each is a directory of files — SiteConfig overrides, sample content, vertical-specific attribute extensions, payment-provider defaults — and an init-vertical script that stamps a fresh project from a chosen preset.

01 — Prerequisites

Before you start this phase

  • Phase 3 closed, tagged v0.4.0. Both providers green, webhooks idempotent.
  • You have at least one real-world reference per vertical — an existing site or operator you can point at and say "this is what this preset should look like when it lands". For most of us, that's askanddeliverwebapp's existing clients.
  • You've re-read the concept brief's go-to-market section. The order of verticals is itself a thesis.

Presets are the product. The template is plumbing; the verticals are what you sell. The single most consequential decision in this phase is what each preset's opinionated defaults are. Spend the time.

02 — What this phase produces

Deliverables

verticals/ folderpresets live here

One subfolder per vertical: verticals/bar-restaurant/, food-truck/, farm-source/, screen-printer/. Each ships site-config.preset.ts, seed/ (sample content, products), and README.md.

Per-vertical attribute extensionsleveraged Product.attributes

Each preset declares its productAttributesSchema (e.g. bar-restaurant adds { allergens, spiceLevel }; screen-printer adds { printSides, garmentBlank }). The Product schema's existing Mixed attributes field accepts these; no schema change.

init-vertical scriptstamps a tenant

npm run init:vertical -- --preset=food-truck --tenant=greenline. Reads preset, applies SiteConfig, seeds DB for that tenantId, sets payment-provider default. Idempotent.

Payment-provider defaults per verticalopinionated

bar-restaurant → Square (POS). food-truck → Square. farm-source → Stripe (online-first). screen-printer → Stripe. Documented as a default, not a hard rule.

Sample content per verticalcredibility on day one

Each preset's seed/ has 5-10 real-feeling Products, 2-3 Pages (Home, About, Menu/Catalog), 1 sample FormDefinition. No lorem ipsum.

doctor.ts grows againtenant awareness

If a non-default tenantId is bound (via env BOUND_TENANT_ID), doctor confirms that tenant exists in DB and its preset is known.

04 — Prompts to run, in order

The phase-4 prompt sequence

Prompt 4.1 — Carry forward + decide vertical order

Prompt 4.1 Resolve OPEN_QUESTIONS from Phase 3. Then make one new decision and document it: In what order are we BUILDING the four presets in this phase? We are not building them all at the same time. The first one will reveal flaws in the preset pattern; the fourth will be the cleanest. Pick the order BASED ON: (a) which vertical do we have the most operator knowledge of right now (i.e. an active client engagement); (b) which vertical's preset will most stress-test the pattern (probably farm-source — all five packs). Document the chosen order in verticals/README.md. Each subsequent prompt assumes that order.

Prompt 4.2 — Establish the preset pattern

Prompt 4.2 Create verticals/_preset-pattern.md and verticals/README.md. Pattern: every vertical/<name>/ ships: 1. site-config.preset.ts: exports `preset: Partial<SiteConfig>` merged onto defaultSiteConfig. 2. product-attributes.schema.ts: Zod schema for that vertical's Product.attributes shape. 3. seed/: products.json, pages.json, forms.json (plain data, no code). 4. README.md: who this preset is for, what it assumes, what a fresh project gets. 5. brand-stub.ts: empty ThemeTokens override for now (phase 5 fills it). Verticals/registry.ts exports all four presets keyed by SiteConfig['vertical']. Commit: "phase-4: preset pattern + registry".

Prompt 4.3 — Build init-vertical script

Prompt 4.3 Create scripts/init-vertical.ts. Signature: npm run init:vertical -- --preset=<key> --tenant=<id> [--force]. Steps: 1. Validate preset key against verticals/registry.ts. 2. Validate tenant id ([a-z0-9-]+, no reserved words like 'default' unless --force). 3. Create a Tenant document (we will introduce the Tenant model in phase 7; for phase 4, a placeholder collection is fine — just a `_tenants` collection with { _id: tenantId, preset, createdAt }). 4. For every model, seed the preset's data into that tenantId. Pages, products, FormDefinitions — all stamped with tenantId. 5. Idempotent: re-running with the same args is a no-op (don't double-seed). With --force, wipe and re-seed. 6. Print a summary table at the end: what was created. Add it to package.json scripts. Test by running for each preset against a fresh local DB; confirm seeds show up scoped to the right tenant. Commit: "phase-4: init-vertical script + idempotency".

Prompt 4.4 — Build preset #1 (per order from 4.1)

Prompt 4.4 Build the FIRST vertical preset per the order decided in 4.1. Create: verticals/<preset-1>/site-config.preset.ts verticals/<preset-1>/product-attributes.schema.ts verticals/<preset-1>/seed/products.json (8 entries, real-feeling, NO lorem) verticals/<preset-1>/seed/pages.json (3 entries: home, about, plus one vertical-specific) verticals/<preset-1>/seed/forms.json (1 entry) verticals/<preset-1>/brand-stub.ts (empty ThemeTokens override) verticals/<preset-1>/README.md Then: - npm run init:vertical -- --preset=<preset-1> --tenant=demo-<preset-1> - Boot the app with BOUND_TENANT_ID=demo-<preset-1>. - Walk every pack's main page and confirm the seeded content is what shows up. If you find any place where the preset pattern doesn't accommodate this vertical's reality, STOP and update the pattern first, then continue. Don't fork the pattern. Commit: "phase-4: preset <preset-1>".

Prompt 4.5 — Build presets #2, #3, #4

Prompt 4.5 Repeat the structure of 4.4 for each remaining vertical, one at a time, in the order from 4.1. Per vertical: 1. Build all five preset files. 2. Run init-vertical against a fresh tenant. 3. Walk the live app for that tenant. 4. If the pattern needed any tweak, that tweak applies to ALL existing verticals — go back and update them in the same commit. 5. Commit each vertical separately: "phase-4: preset <name>". DO NOT batch them. We want the diff per vertical to be readable in isolation.

Prompt 4.6 — Vertical-aware doctor

Prompt 4.6 Extend scripts/doctor.ts: - If env.BOUND_TENANT_ID is set: - confirm a _tenants document exists for it. - confirm its preset key matches a registered vertical. - confirm at least one seeded record exists per pack the preset declares "on". If BOUND_TENANT_ID is unset, doctor behaves as before (default tenant). Commit: "phase-4: tenant-aware doctor checks".

Prompt 4.7 — Close the phase

Prompt 4.7 Doc update pass: - Append Phase 4 entry to CHANGELOG with the order-of-build and every preset shipped. - Bump package.json to 0.5.0. - Update repo-context.html §05 file map to include verticals/ and scripts/init-vertical.ts. - Update glossary.html: "preset", "vertical", and the relationship to tenantId. - Update README.html doc-meta version. - List OPEN_QUESTIONS for Phase 5 — especially: which preset's brand identity is closest to "ready" (will become the test case for the visual system)? Tag v0.5.0. Fresh-clone test.
05 — Acceptance criteria

Phase 4 is done when…

  • All four preset folders exist and conform to the preset pattern.
  • npm run init:vertical -- --preset=X --tenant=Y works for every preset, idempotently.
  • Booting the app with BOUND_TENANT_ID=demo-X shows X's seeded content.
  • Switching BOUND_TENANT_ID between two tenants in the same DB shows no leakage (the scoped() investment from Phase 1 pays off here).
  • Each vertical's payment-provider default reads correctly through getPaymentProvider.
  • Tagged v0.5.0.

The big tenant-leakage test. Seed two tenants. Hit one tenant's product list endpoint with the other tenant's bound id. Confirm zero overlap. This is the cheapest insurance you'll ever buy against a phase-7 horror story.

06 — Status & handoff to Phase 5

What's true at the end of Phase 4

The template is no longer generic — it's four opinionated SaaSes that happen to share 70% of their code. Phase 5 is where they start to look as different as they should.

What Phase 5 will pick up

  • The four-layer token system from the visual-system brief: foundation → vertical → tenant pick → tenant override.
  • The contrast validator + the leaf-only override rule.
  • CSS variable emission so the same React app renders as four (then many) visually distinct sites.
Open question to carry forward

Each preset shipped a brand-stub.ts that is empty. Phase 5's first prompt has to decide: do we ship a opinionated brand default per vertical (e.g. food-truck = bold, high-contrast, primary brand red), or stay neutral and force every client to pick? The concept brief leans opinionated; confirm before phase 5.
