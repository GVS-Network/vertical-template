Rules Cursor Rules Overview · v0.1.0

The rules layer.

What's in .cursorrules, what's in .cursor/rules/*.mdc, when each one fires, and how to edit them without accidentally weakening the system.

01 — Two rule systems

Why both .cursorrules and .cursor/rules/

Cursor supports two rule mechanisms simultaneously. The repo uses both because they cover different cases:

.cursorrules legacy · single file · always-on

A single plaintext file at repo root. Loaded into every chat session, every file. Best for short, absolute principles that must always be visible. Our copy is ~100 lines and stays that way.

.cursor/rules/*.mdc scoped · multi-file · conditional

Folder of .mdc files (Markdown with frontmatter). Each file has a description, optional globs, optional alwaysApply. Cursor loads each file when the right context is in scope — typically when an editor file matches one of its globs.

i

When they conflict. The more specific (glob-scoped) rule wins. .cursorrules is the floor; the .mdc files refine for context. If you find yourself writing the same rule in both, delete the .cursorrules copy.

What's installed and what each one does

Always-applied (load on every chat)

00-project-context.mdc

The thesis in one screen. Names the three briefs, the codebase family, the workflow. The "primer" — what Claude needs in working memory regardless of which file is open.

70-multi-tenant-seams.mdc

The two seams (getSiteConfig, getPaymentProvider) and the tenantId rule. Always on because the discipline applies everywhere — not just inside server/.

80-versioning-and-docs.mdc

Two versions (template + docs). Doc-update triggers per change-type. CHANGELOG protocol. Brief-revision procedure.

99-claude-collaboration.mdc

How Claude should pace, what to flag, what to ask before doing. The "voice" rule — keeps sessions from going sideways.

Glob-scoped (load when matching files are open)

10-typescript-conventions.mdc **/*.ts · **/*.tsx

Strict mode, no any, zod for runtime validation, import grouping, file naming. The "before you write TS" rule.

20-server-conventions.mdc server/**/*

Express patterns, Mongoose patterns, the seam rules in their server-specific form, feature-pack toggle protocol.

30-client-conventions.mdc client/**/*

React + Vite patterns, routing, CSS variable consumption rules, site.config.ts / site.theme.ts as the only legitimate per-build files, Custom*.tsx protocol.

40-feature-packs.mdc **/features/**, **/providers/**

Anatomy of a pack, the toggle protocol, cross-pack isolation, the two-or-more-verticals test.

50-siteconfig-contract.mdc **/types/site-config.ts · **/types/theme-tokens.ts

Twin-file contract. Editing protocol. What can and cannot go in the type files.

60-visual-system.mdc **/theme/**, **/site.theme.ts, **/*.css, **/*.module.css

Four-layer resolver. Leaf-only overrides. Registry expansion via "second use." No hardcoded design values outside theme/.

03 — Anatomy of a rule

How an .mdc file is structured

---
description: One-line summary shown in the rule picker.
globs:
  - "server/**/*"
  - "shared/server/**/*"
alwaysApply: false           # default; can be omitted
---

# Rule Title

Markdown content. Cursor passes this to the model when the rule is active.

Keep it under ~200 lines. Long rules get ignored or compressed.

Reference other docs with relative paths so Claude can open them:
- Full reference: `docs/rules/documentation-maintenance.html`.

Conventions for our rules

  • Numbered prefixes. 00- through 99-. Lower numbers are "earlier in the mental stack" — project context, then conventions, then meta-rules. Cursor doesn't actually order them, but humans editing the folder benefit.
  • Description is a sentence, not a phrase. "TypeScript conventions for any .ts/.tsx file in this repo" — not "TS rules."
  • Globs are specific. Prefer server/**/* over **/*.ts when the rule is server-specific. Overlap is fine; vagueness isn't.
  • One rule, one file. If a rule grows past ~200 lines, split it. Long rules are skipped silently by the model.
  • Cross-link rather than duplicate. If rule 60 wants to remind about the contract-check script, link to rule 50 — don't restate the rule.
04 — Editing rules

When to edit, and how

✓

Edit a rule when the same correction has been given more than twice. If you've told Claude the same thing three times across three sessions, the rule isn't catching it. Either tighten the wording, add a clearer example, or add a new rule with a tighter glob.

✓

Edit when a phase prompt completes. A completed phase often reveals a rule that should have existed from the start. Phase docs include a "rules-to-write" check at the end.

✓

Edit when a brief revises. Brief revisions usually map to rule revisions. The doc-maintenance flow flags this.

!

Don't weaken a rule because a single session pushed back. Rules exist because of patterns, not single instances. If a rule feels wrong in one session, write the friction down and revisit the rule between sessions, not during one.

!

Don't add a rule for a single client's edge case. Client-build behavior lives in site.config.ts and site.theme.ts. The rules govern the platform; they should generalize across builds.

05 — Versioning rules

How rule changes are tracked

Rule files live in the repo and version with the code. A rule change is a normal commit; major rule restructures bump the docs version in docs/README.html.

Patch no docs bump

Wording fixes, typo, example added, glob narrowed without behavior change. CHANGELOG entry under Docs.

Minor docs minor bump

New rule added. Existing rule expanded. New glob added. Behavior of an existing rule tightened. CHANGELOG entry under Docs; this README's version increments.

Major docs major bump

Rule retired. Rule restructured (split into two, merged from two). Rule's behavior weakened in a way clients should know about. CHANGELOG entry under Docs with prominent note; usually corresponds to a brief revision.

Retiring a rule

Don't delete the file. Move it to .cursor/rules/retired/ with a header note explaining what replaced it and on what date. Keeps the rule history searchable; explains future "why was this not enforced?" questions.

06 — Sanity-check the rules

The five-minute audit

Run this informally every few weeks, formally at each minor template release:

  1. Open Cursor on a representative file (e.g. server/src/features/intake/routes/submit.ts). Check the rules indicator in the chat sidebar — it should show the always-applied set plus the matching scoped rules. Wrong glob → fix the glob.
  2. Read each .mdc file end-to-end. Anything stale (references to a feature that no longer exists, a vertical that was renamed, a phase that completed)? Patch it.
  3. Compare .cursorrules to the .mdc set. Duplicates? Delete from .cursorrules. Contradictions? Resolve.
  4. Look at the last ten PRs. Any of them required a Claude correction more than once? That's a rule gap.
  5. Look at the open questions in the briefs and contexts. Any of them resolved silently in code without an answering rule? That's a missing rule.
07 — Status

The rule set as of v0.1.0

One .cursorrules + ten .mdc files: four always-applied, six glob-scoped. The set is intentionally minimal — every rule corresponds to a specific failure mode the briefs anticipated. Adding rules is easy; removing or changing them later is harder, so additions are deliberate.

Anticipated rule changes

Q · Phase-7 rules

When multi-tenancy ships, rule 70 ("seams") and rule 20 ("server") get substantial revisions — the single-tenant assumption that's currently load-bearing relaxes. Plan a doc-major bump.

Q · Per-client rule overlay

Per-client builds may want a small additional rule set forbidding edits to core files. Candidate: a client-build/.cursor/rules/00-type-c-only.mdc shipped with each build at phase 6.

Q · Skill-attached patterns

Recurring prompt patterns may graduate into Cursor "skills" or scoped rules with file attachments. Track candidates in CHANGELOG.
