Rules Documentation Maintenance · v0.2.0

Docs that keep up with code.

The versioning model, the change-types for docs, and the triggers that force a documentation update before a PR can merge.

01 — The premise

Why this rule exists at all

The three project briefs and the contexts under docs/ are not "nice to have." They are how Claude (and future-you) recovers context after time away. A code change that lands without its corresponding doc edit creates an invisible debt — the system's memory of why drifts out of sync with what the code now does.

One-line framing

Code without docs is a guess that the next person will infer the same thing twice. The rule is simple: every code change either is covered by existing docs, or updates docs in the same PR, or surfaces a brief revision. There is no fourth option.

02 — Two versions

Template version vs. docs version

Template version CHANGELOG.md · semver

The version client builds pin to. Lives in CHANGELOG.md at repo root. Bumps on tagged releases of the template.

MAJOR — breaking change to public surface (SiteConfig shape, ThemeTokens shape, feature pack API). Migration steps required.

MINOR — new feature pack, new vertical preset, new payment adapter, new registry entry. Backward compatible.

PATCH — fixes, internal refactors, additive Type-B changes that existing builds opt into.

Docs version README.html header · semver

The version of the documentation family. Lives in docs/README.html's header (Build Documentation · v0.X.Y).

MAJOR — doc retired, doc restructured, a brief revised in a way that changes thesis-level claims.

MINOR — new doc added, new rule added, new section in an existing doc that captures a new pattern.

PATCH — typos, clarifications, examples, wording polish.

The two version numbers are independent. A docs-only PR doesn't bump the template version. A template bug fix doesn't bump the docs version (unless the fix corrects something the docs said).

03 — CHANGELOG protocol

The shape of CHANGELOG.md

Keep-a-Changelog style. Headings in this order under each release:

## [Unreleased]

### Added
- Feature packs scaffolding for catalog and content.

### Changed
- Auth0 middleware moved into auth feature pack (`server/src/features/auth/`).

### Fixed
- IntakeSubmission missing tenantId index (compound w/ submittedAt).

### Thesis
- (rare) Brief revision · template-architecture-brief §03 decision row 4
  updated to reflect that auth lives in features/, not middleware/.

### Docs
- Added rules/cursorrules-overview.html.
- Updated stack-context §03 to add `zod-form-data` to allowed deps.

## [v0.4.2] — 2026-XX-XX

(consolidated from [Unreleased] at release tagging)

Conventions

  • Every PR appends to [Unreleased]. Release tagging is the only event that converts [Unreleased] into a versioned heading.
  • Sections are added on demand. A PR with no Thesis change doesn't include the heading.
  • Entries are user-facing. "Refactored intake handler" is too internal; "Reduced intake submission latency from 800ms to 200ms" is real news.
  • Link to the PR. Each bullet ends with a parenthesized PR number once the PR is opened.
04 — The update matrix

Which doc must update for which change

New top-level folder

contexts/repo-context.html §05 file map. CHANGELOG.

New dependency

contexts/stack-context.html §03 allowed list. CHANGELOG.

New env var

contexts/stack-context.html §04. .env.example on the relevant side. CHANGELOG.

New SiteConfig field

Both type files (contract:check enforced). template-architecture-brief.html §02 if the new field is structurally significant. contexts/glossary.html if the field has a semantically loaded name. CHANGELOG.

New feature pack

template-architecture-brief.html §03 decision row and §04 verticals if it changes the pack count. contexts/glossary.html §04 (canonical packs). New rule under .cursor/rules/ if the pack has unusual conventions. CHANGELOG.

New vertical

platform-concept-brief.html §01 client-landscape table. template-architecture-brief.html §04. Vertical SiteConfig literal in the type. Preset files. Theme preset file. Phase-4 prompt update. CHANGELOG.

New payment provider

template-architecture-brief.html §03 PaymentProvider row. contexts/stack-context.html §01. The provider's adapter file under server/src/providers/. CHANGELOG. (The first third provider is a Thesis-level event.)

New token category

visual-system-brief.html §03 registry shape, §04 cards. Type file (both sides, contract-checked). Resolver. Validators. CHANGELOG under Thesis if the category is foundational, Added if additive-optional.

New registry entry (palette, pairing, preset)

The registry file. visual-system-brief.html only if the addition signals a pattern (third palette in a vertical that previously had one). CHANGELOG under Added.

Phase prompt completion

The phase doc's acceptance criteria checked off. Any acceptance criterion that turned out wrong gets revised in the same PR. Next phase's "Prerequisites" sanity-checked. CHANGELOG.

Rule change (any .cursor/rules/*.mdc edit)

rules/cursorrules-overview.html §02 if the rule was added, retired, or substantially restructured. CHANGELOG under Docs.

Brief revision

The brief's version in its header bumps. docs/README.html docs-version bumps. Cross-brief and context refs flagged in the same PR. CHANGELOG under Thesis.

05 — CI enforcement

What the build checks automatically

Two scripts run in CI to keep docs honest. They catch the obvious cases; reviewers catch the subtle ones.

doc-changelog-check

Asserts that any PR with a code change has at least one new line under [Unreleased] in CHANGELOG.md. Pure-comment-only and pure-typo PRs can override with a [skip changelog] label.

doc-coupled-files-check

A static map of file-coupling. If a PR changes files matching pattern A without changing files matching pattern B, it fails. The map (starts here, expand as needed):

# Each entry: changed-path-glob -> required-also-changed-glob
package.json                        => docs/contexts/stack-context.html
.env.example                        => docs/contexts/stack-context.html
**/types/site-config.ts             => docs/contexts/glossary.html | docs/template-architecture-brief.html
**/types/theme-tokens.ts            => docs/visual-system-brief.html
verticals//preset.ts           => docs/template-architecture-brief.html
verticals//theme.preset.ts     => docs/visual-system-brief.html
server/src/providers/.ts       => docs/template-architecture-brief.html
.cursor/rules/**                    => docs/rules/cursorrules-overview.html
docs/*-brief.html                   => docs/README.html  # docs version bump

The pipe means "one of these". The check is a heuristic, not a proof — overrides exist via the PR label [docs-checked] when the reviewer has explicitly verified the docs are correct without the coupled file changing.

06 — Brief-revision protocol

The deliberate path for thesis edits

Briefs are not casually edited. The friction is on purpose — every brief edit is a strategic claim about what the platform is and why.

01Surface

Open the question explicitly

Write down what's changing, what current claim it contradicts, and what triggered the revision. This is the PR description's first paragraph. Phrase it as a decision, not a fix.

02Scope

Identify dependents

Which other briefs reference the changed section? Which contexts? Which rules? Which phase prompts? Often the changed section is internally cross-linked elsewhere — find those refs and either fix them in the same PR or open follow-ups.

03Edit

Make the change with version bumps

Edit the brief. Bump its header version (v1 → v2). Bump docs/README.html's docs version (usually minor; major if the revision changes a foundational claim).

04Log

CHANGELOG entry under Thesis

One paragraph: what the old claim was, what the new claim is, why the change. Reference the PR. Future-you reading the CHANGELOG should be able to reconstruct the decision without re-reading the brief.

05Settle

Update the rules

If the brief revision contradicts a rule, the rule needs updating in the same PR. If the brief revision creates a new rule (e.g. "we no longer support X"), the new rule lands in the same PR.

07 — Anti-patterns

What this rule explicitly forbids

!

Silent brief edits. Editing a brief without bumping its version, without a CHANGELOG entry, or without checking dependents. The briefs are the system's memory; quiet edits to them are how the memory gets corrupted.

!

"Update docs later" deferrals. Doc debt compounds. The PR that adds the code without the doc is the one with the context; any later attempt to write the docs will miss something. Always same-PR.

!

Speculative docs. Don't document features that don't exist yet. Phase prompts describe what's coming; briefs describe what's decided. New context entries are for things that are real now.

!

CHANGELOG as a dumping ground. If the entry would be more useful in the relevant context or brief, put it there. CHANGELOG is for chronology — what changed, when. The context is for state — what is true.

08 — Status

What this rule enforces today

v0.1.0 — the matrix, the CI checks (specified, not yet implemented in the template's scripts/), the brief-revision protocol. The matrix is the load-bearing part; the CI is its enforcement. Without the CI, the matrix becomes informal review pressure, which works for a small team and degrades as the team grows.

Open questions

Q · Implementing the CI scripts

Phase 0 lands the doc-changelog-check; doc-coupled-files-check follows once the docs/ folder exists in the template repo. Until then, the matrix is reviewer-enforced.

Q · Per-client doc divergence

Phase 6 introduces client builds. Do client builds maintain their own build-notes.html? If yes, this rule needs a section on per-build vs. platform docs.

Q · Skill-attached docs

If recurring patterns graduate into Cursor "skills," the matrix needs a row for skill-file changes.
