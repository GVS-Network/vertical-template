Visual System Companion to Template Brief · v0.1

A token registry with guarded override seams.

The customization layer that turns the template brief's branding slice into something that can absorb color, type, logos, blocks, density — and the categories not yet named — without breaking client builds already in production.

01 — Premise

What this brief is solving for

The template brief established that branding is per-tenant config, not core code, and that the SiteConfig.branding slice holds { name, tagline, theme, fonts, contact }. That's enough to launch a build. It is not enough to evolve.

The pressure comes in three forms: (1) clients will ask for things their preset doesn't anticipate — a specific brand red, a custom display font, a denser hero block; (2) the four verticals will diverge visually as they mature, and the differences need somewhere to live; (3) new categories of visual choice — motion later, dark-mode variants, accessibility presets — will arrive and need a home that doesn't disrupt builds already shipped.

One-line framing

The visual system is a token registry with three layers of resolution (defaults → vertical preset → tenant overrides) and explicit override seams at every layer. New categories enter the registry as additive fields with safe defaults, propagate as Type B, and never force a re-render of existing builds until the tenant opts in.

Position relative to the existing briefs

This is not a new system. The concept brief named branding as tenant-config; the template brief gave it a field on SiteConfig. This brief is the schema and discipline for what goes inside that field once there's more than one client to keep coherent — and once the categories you'd want to add later are anticipated by the shape.

02 — Architecture

The four-layer resolution model

At runtime, every visual value a component reads — a color, a radius, a font stack — resolves through the same four layers in the same order. Each layer is optional except the first. The resolver is one function; the layers are data.

L1 · Foundation
Platform-wide defaults. Every token in the registry has a value here. Existing builds inherit changes only when they pull and rebuild — but the values are conservative enough that visible drift is rare. Type-A propagation.
tokens.foundation.ts
L2 · Vertical
Per-vertical opinion. Farm-source ships earthy; screen-printer ships graphic; bar-restaurant ships warm. Overrides L1 selectively. Lives next to the existing vertical preset.ts as theme.preset.ts.
verticals/farm-source/
theme.preset.ts
L3 · Tenant choice
Tenant selects from the registry — palette ID, type-pairing ID, density preset, block-style preset. The guided-customization rail. No raw values yet; just picks from sets the system already knows.
site.theme.ts
// palette: 'rust-clay'
L4 · Tenant override
The escape hatch. Specific token values the tenant wins on — exact hex, custom font file, hand-tuned radius. Validated against a guardrails schema (contrast, readable size, etc.) so overrides can't break accessibility or the layout grid.
site.theme.ts
// overrides: { color.brand: '#7A1F1F' }
✓

The resolver is the seam. One function, resolveTokens(tenantId), walks the four layers and emits the final CSS variable set. Components never know which layer a value came from. Adding a fifth layer later (per-page overrides, A/B variants) is a function change, not a component change.

03 — Registry

The shape of themeTokens

One TypeScript type, mirrored client and server (same contract-check pattern as SiteConfig). Every category is namespaced. Every leaf is a value the resolver can produce.

interface ThemeTokens {
  color: {
    brand:       { primary, secondary, accent };
    surface:     { page, panel, raised, sunken };
    ink:         { default, soft, muted, inverse };
    semantic:    { success, warning, danger, info };
    palette:     PaletteId;       // L3 pick
  };
  typography: {
    pairing:     PairingId;       // L3 pick
    display:     { family, weight, scale };
    body:        { family, weight, scale };
    mono:        { family, weight };
    scale:       'compact' | 'standard' | 'spacious';
  };
  logo: {
    primary:     AssetRef;        // light bg
    inverse:     AssetRef;        // dark bg
    mark:        AssetRef;        // square / favicon source
    favicon:     AssetRef;
    social:      AssetRef;        // 1200x630 OG
  };
  blocks: {
    card:        StylePresetId;   // 'soft', 'sharp', 'editorial', 'panel'
    image:       StylePresetId;   // treatment, ratio, mask
    section:     StylePresetId;   // rhythm + separators
    button:      StylePresetId;
  };
  layout: {
    density:     'comfortable' | 'standard' | 'compact';
    radius:      RadiusScale;
    spacing:     SpacingScale;
    container:   ContainerScale;
  };
  motion?: MotionTokens;          // reserved, not v1
}

Two properties are worth stating explicitly. L3 picks live alongside L4 overrides in the same type — palette: 'rust-clay' and brand.primary: '#7A1F1F' coexist; the resolver applies the palette first, then the overrides win. Every category that v1 doesn't yet need is reserved by name with an optional type — motion? is there so the schema doesn't break when phase-2 motion lands.

→

The registry is the rail. A tenant can pick any PaletteId the registry exports; they cannot invent one. A tenant can override brand.primary to any hex; they cannot add a new color slot. The categories are open to evolution by the platform; the values within them are open to override by the tenant.

04 — The five categories

What v1 ships and what v1 anticipates

Five categories you named as near-term, each modeled the same way: a registry of named presets, a guardrails schema for raw overrides, and a vertical default per preset.

color
Palettes & semantic roles

Palette IDs map to brand/surface/ink/semantic role groups. Tenants pick by ID; overrides hit specific roles. Contrast guardrails enforce WCAG AA on resolved pairs — if a tenant overrides ink.default against surface.page and the ratio falls below 4.5:1, the resolver throws at build, not in production.

L1 · neutral, accessible base
L2 · per-vertical palette
L3 · palette ID pick
L4 · role-level hex override
logo
Brand marks & variants

Five asset slots covering light bg, dark bg, square mark, favicon source, social OG. AssetRef is Cloudinary-backed (same as the agency app's PortfolioProject uploads). Mark and favicon auto-derive from a single upload if the tenant doesn't provide variants; explicit variants override.

slots · 5 (primary, inverse, mark, favicon, social)
derivation · auto from mark when absent
storage · Cloudinary, agency-app pattern
typography
Pairings, scale, weight

Type pairings ship as named sets (display family + body family + suggested scale). Custom fonts are an override: tenant supplies WOFF2 files, the system writes @font-face into the resolved CSS. Scale is a token group, not a constant — compact shaves 2pt off the type ramp without touching individual sizes.

blocks
Content block style presets

The category that does the most quiet work. Cards, image treatments, sections, buttons each have a small set of style presets ('soft' / 'sharp' / 'editorial' / 'panel' for cards, for instance). Shared block components consume the resolved preset; tenants pick a set per block type.

Density & spacing scale

One density choice cascades through spacing and container scales — comfortable, standard, compact. Radius scale is its own token group (sharp / soft / pillowy) since it doesn't covary cleanly with density. Most tenants will touch density and nothing else here.

05 — Override discipline

Guided customization, in practice

Guided customization means: tenants get rails, and the rails have known places where you step off them. Three rules govern every override.

Overrides are leaf-level only no shape drift

A tenant can override color.brand.primary. A tenant cannot add color.brand.tertiary. The category shape is owned by the platform; the leaf values are negotiable. This is what keeps the resolver from becoming a merge engine for arbitrary JSON.

Every override is validated accessibility & coherence

The themeOverrides validator runs at build time. Contrast pairs checked. Custom fonts checked for required weights. Density choices checked against block-preset compatibility. A failing validation is a build error — the tenant's site falls back to L3 with a logged warning.

Overrides are auditable tenant-visible diffs

The resolved token set is written to .theme-resolved.json per build. A tenant (or the operator on their behalf) can see exactly which values came from which layer. When the platform's L1 shifts in a future release, the diff is one file.

The CMS-shaped path is not v1 honest scope

Editing site.theme.ts is a code edit, not a UI action. A tenant-facing theme editor is its own product surface and follows the template brief's "no CMS UI yet" discipline. When the patterns crystallize across builds, the editor graduates back into core.

06 — Propagation

How new visual categories reach existing builds

Type-B for the category, Type-A for the schema. This is the rule that makes "evolve the system without rebuilding the clients" actually true.

Type A · automatic
The schema, the resolver, the validators

Adding the motion? field, expanding role names inside color, shipping a new block-preset option. Lands via git pull. Safe because the field is optional and the default produces the existing rendering.

Type B · advisory
The defaults, the preset library

A new palette, a new pairing, a new vertical preset value. Available to opt into; existing builds keep their explicit picks. Release notes flag them as available.

Type C · never
The tenant's site.theme.ts

Per-build divergence — chosen IDs, raw overrides, custom assets. Upstream pulls do not touch this file. Once multi-tenant ships (phase 2 of the template brief), this collapses into tenant documents.

i

The motion example, played out. Phase-2 ships motion?: MotionTokens: schema change (Type A — every build gets the field), three motion presets in the registry (Type B — available, not applied), and a vertical default of undefined (no change to anyone's rendering). A client says "let's add scroll reveals." Their site.theme.ts sets motion: 'subtle'. One file, one build. No other client moves.

07 — Worked example

A farm-source client picks rails, then steps off one

How a real configuration lands in the codebase. The client is a small CSA operation; the preset gets them 80% there; the brand red is non-negotiable; everything else stays on rails.

Build flow

1
Init. npm run init:vertical -- --vertical=farm-source --tenant=blackoak. Drops a site.theme.ts seeded with the farm-source theme preset — earthy palette, serif/sans pairing, comfortable density, soft cards.
2
L3 picks. Client likes the preset, swaps the palette: palette: 'clay-and-cream' instead of the default 'field-and-loam'. Switches block.card to 'editorial'. Everything else inherits.
3
L4 override. Their brand uses a specific deep red. They add overrides: { 'color.brand.primary': '#7A1F1F' }. Validator checks contrast against resolved surface.page — passes.
4
Logo. Tenant uploads a single SVG mark. The system derives favicon and social automatically; primary and inverse variants are opt-in if the brand has them.
5
Build. Resolver walks L1→L2→L3→L4, emits :root CSS variables, writes .theme-resolved.json. Total tenant-authored file: 14 lines. Total upstream-pullable code: untouched.
6
Six months later. Platform ships motion. Their resolved theme gains motion: undefined automatically. Their site renders identically. They can opt in when ready.
08 — Discipline

What this layer requires of the operator

→

Register every category before tenants ask for it. The moment a category lives only in one client's overrides, it has skipped the registry — and the next client who asks for "the same thing" can't get it without a code change. When you see a second override of the same shape, lift it into L1.

→

Preset libraries grow slowly, deliberately. Three palettes per vertical, not thirty. Two pairings, not fifteen. The Squarespace error is infinite choice; the wedge is opinionated subsets. Add a preset when you've used it twice, not when you've imagined it once.

→

Treat the validator as a contract. When a client wants an override the validator rejects (a specific red that fails contrast on a specific surface), the answer is "adjust one side, not the validator." The validator is what keeps the platform's promise that no build is broken by configuration.

→

Resist motion in v1. The schema reserves it; the implementation defers it. Motion done wrong is the fastest way to make several different verticals start looking alike. It earns its place after the static system has stabilized across the first cluster of clients.

09 — Status

Where this brief leaves the system

The visual customization layer is a token registry with four resolution layers, guarded overrides at the leaf level, and additive evolution of categories via Type-B propagation. Five categories land in v1 — color, logo, typography, blocks, layout. Motion is named in the schema and deferred in the implementation. The resolver is one function; the discipline is one rule (register before second use).

Open questions deferred to subsequent threads

Q · Asset pipeline

Cloudinary direct (matches agency app) or a per-tenant prefix? When a logo gets uploaded via the tenant's eventual admin UI, what's the path from upload to AssetRef?

Q · Custom-font hosting

Tenant uploads WOFF2 — stored where, served from where, and how does @font-face get into the resolved CSS at build time vs. runtime?

Q · Validator strictness

WCAG AA is the floor; is AAA an opt-in per vertical (farms and food might want stricter outdoor-readable defaults)? What does "fail the build" mean in practice for a tenant who is mid-edit?

Q · Block-preset coupling

Card style 'editorial' assumes a certain section rhythm. When tenants mix presets that don't compose, does the validator catch it, or do we ship visual debt and learn?

Q · Theme editor surface

When the editor graduates from code to UI, does it live in the operator's admin (gated, paid) or in each tenant's own admin (self-serve)? The answer probably tracks the concept brief's graduation criteria.
