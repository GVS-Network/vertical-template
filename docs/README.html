README Build Documentation · v0.6.0

The build documentation family.

An index of every doc that drives the Cursor + Claude build loop, the order they're read in, and the rules that keep them honest as the system evolves.

01 — Purpose

What this folder is for

The three project briefs (concept · template · visual system) describe what is being built and why. This folder describes how the build actually happens: which AI agent reads what, in which order, with which rules in force, and how the docs themselves stay current as the codebase changes underneath them.

One-line framing

The briefs are the thesis. The build docs are the operating system. Cursor + Claude is the runtime. Every line of code in the template should be traceable through these docs back to a decision in a brief.

Before npm run dev, run npm run doctor from the repo root. It verifies Node >= 20, that every key in each .env.example exists in the matching .env (root, server, client), and that MONGODB_URI connects within 3 seconds. It does not validate Auth0 or payment credentials — only presence and Mongo reachability.

04 — How to actually use this

The build loop

01Open

Open Cursor on the repo

Cursor auto-loads .cursorrules and every .cursor/rules/*.mdc file that matches the files you have open. You don't activate them — they activate themselves. Verify the indicator in the chat sidebar shows the rule count.

02Prime

Paste the session starter

Open contexts/session-starter.html, copy the prompt block at the bottom, paste into Cursor's chat. It tells Claude which phase you're in, which briefs are active, what's already built, what is forbidden.

03Phase

Pick the phase prompt

Open the prompts/phase-N-*.html for the current phase. Each contains: prerequisites, deliverables, prompts to feed Cursor in order, and acceptance criteria.

04Loop

Prompt → review → commit → next prompt

Each phase doc breaks the work into prompts small enough that Claude can complete one and you can review the diff before the next. Don't batch them.

05Close

Update docs before merging

The doc-maintenance rule determines what must change. CHANGELOG and any affected brief/context update in the same PR as the code.

05 — Versioning

How these docs version, and why it matters

This documentation set carries two versions: a doc-set version (this file's v0.6.0 in the header) and a template version (CHANGELOG.md at repo root). They are not the same.

Template version what client builds pin to

Semver. v0.4.1 → v0.4.2. Tagged on the template repo. Client builds pin to a tag and bump deliberately. This is the version the template brief refers to in its upgrade-flow rule.

Docs version documentation lineage

Independent semver in this README's header. Bumps when the doc family adds, retires, or restructures a doc. Track in CHANGELOG under a docs heading. A docs-only change doesn't bump template version; a template change that needs new docs bumps both.

Brief revisions thesis edits are explicit

The three briefs carry their own v1, v0.1, v0.1 in their headers. Editing a brief is a deliberate act — it means the strategy changed. Brief edits always bump docs version and demand a CHANGELOG entry under thesis.

→

Briefs are not retroactively edited to match code. If the implementation diverges from a brief, either change the code back, or write a new brief revision and explain the decision. The briefs are the system's memory of why — protect that.
